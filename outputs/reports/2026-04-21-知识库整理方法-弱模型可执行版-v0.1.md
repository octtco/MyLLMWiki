# 问题

怎样把资料稳定整理进这座知识库，而且细到换一个稍弱的模型来做，也还能做出差不多的效果？

## 短答案

关键不是让模型“发挥聪明”，而是把它锁进一套固定动作里。

这套固定动作至少要包含：

1. 固定的分层结构  
2. 固定的页面类型  
3. 固定的写作顺序  
4. 固定的证据强度规则  
5. 固定的页面模板  
6. 固定的验收清单  

一句话说：

**不要让模型自由发挥整理知识库，要让它按规程施工。**

## 展开

## 一、这套方法要解决什么

当前最核心的问题不是“能不能写出几篇笔记”，而是：

- 能不能把资料变成可调用的知识结构
- 能不能稳定长出主题 / 方法 / 概念 / 案例 / 来源
- 能不能同时给人和 agent 用
- 能不能减少“换个模型就换一套口径”的波动

所以这套方法的重点不是文风，而是控制流程。

## 二、知识库的最小结构约定

整理前，先把仓库理解成 4 层：

### 1. 原始层 `raw/`

放原始资料，只读，不往里写知识性总结。

### 2. 编译层 `wiki/`

放正式知识页。

当前最小页面类型：

- `wiki/sources/`
- `wiki/topics/`
- `wiki/methods/`
- `wiki/concepts/`
- `wiki/cases/`
- `wiki/domains/`
- `wiki/relations/`

### 3. 结果层 `outputs/`

放：

- 章节工作层
- 阶段性报告
- 方法提炼
- 还不够稳定、暂不回写的内容

### 4. 控制面 `_index/`

放：

- 总览
- 索引
- 待处理
- 日志
- 健康检查
- 维护协议

## 三、模型进入任务后先读什么

弱模型一上来不要直接写页，先固定读这些文件：

1. `_index/总览.md`
2. `_index/维护协议.md`
3. `raw/README.md`
4. `wiki/README.md`
5. `outputs/README.md`
6. 与当前任务最相关的索引页

如果是基于已有样板继续做，再额外读：

- 最近的对齐页
- 当前来源页
- 当前主题页
- 当前最相关的方法页和概念页

## 四、默认主路径

这座库当前默认不是“来源 -> 概念”的直连，而是：

`领域页 -> 主题页 -> 方法页 -> 概念页 / 案例页 -> 来源页`

你可以把它理解成：

- 领域页：去哪找
- 主题页：我在处理什么问题
- 方法页：可以怎么做
- 概念页：怎么理解和表达
- 案例页：具体长什么样
- 来源页：这些东西从哪来

弱模型的任务，不是打乱这条路径，而是尽量把新资料接到这条路径里。

## 五、整理任务的统一流程

下面这套流程建议写死给模型。

### 第 0 步：先判断这是哪类资料

至少先判断：

- 短资料
- 长文 / 长书
- 多来源同主题资料

不同类型走不同节奏。

### 第 1 步：先判断这次整理目标

目标可能是：

- 先建来源页
- 先做一个主题样板
- 先抽方法
- 先做章节级工作层

没先定目标，后面很容易页面乱长。

### 第 2 步：先建或更新来源页

来源页是总入口和提炼起点。

不管后面长多少页，来源页都应先有。

### 第 3 步：判断是否需要工作层

如果是长书、长报告或高密度资料，必须先走 `outputs/` 工作层，不要直接进 `wiki/`。

### 第 4 步：抽候选池

先抽：

- 候选主题
- 候选方法
- 候选概念
- 候选案例

### 第 5 步：筛选正式页

不要把候选池直接全建成页面。

### 第 6 步：按固定顺序落页

更稳的顺序：

1. 来源页  
2. 主题页  
3. 方法页  
4. 概念页  
5. 案例页  
6. 索引与日志

### 第 7 步：做显式链接

每个新页面至少要满足：

- 能从一个索引页进入
- 能连到一个上游页
- 能连到一个平级或下游页

### 第 8 步：更新控制面

只要这次改动影响结构或可发现性，就更新：

- `_index/总览.md`
- 对应索引页
- `_index/log.md`

## 六、每类页面的最低模板

这一部分最好写死。模型照模板填，波动会小很多。

## 1. 来源页最低模板

### frontmatter 最低字段

- `type: source`
- `status`
- `source_path`
- `source_type`
- `title`
- `updated_at`
- `tags`
- `related_topics`
- `related_methods`
- `related_concepts`
- `related_cases`

### 正文最低结构

1. 一句话总结  
2. 当前整理范围  
3. 摘要 / 前两章直接在做什么  
4. 为什么值得提炼  
5. 证据或原文线索  
6. 当前不宜说死的结论  
7. 已拆出的概念 / 方法 / 案例  
8. 相关主题 / 关系  
9. 回链

### 来源页禁止事项

- 不要把所有正式知识都塞在来源页里
- 不要只写摘要
- 不要不写边界

## 2. 主题页最低模板

### frontmatter 最低字段

- `type: topic`
- `status`
- `related_methods`
- `related_concepts`
- `related_cases`
- `source_pages`

### 正文最低结构

1. 问题定义  
2. 为什么重要  
3. 当前结论  
4. 判断框架  
5. 相关方法  
6. 关键概念  
7. 代表案例  
8. 相关来源  
9. 还不清楚的地方  
10. 后续可追的问题

### 主题页禁止事项

- 不要只做链接堆
- 不要只写成概念词典
- 不要没有“当前问题是什么”

## 3. 方法页最低模板

### frontmatter 最低字段

- `type: method`
- `status`
- `related_topics`
- `related_concepts`
- `source_pages`

### 正文最低结构

1. 这个方法是什么  
2. 适用于什么情况  
3. 大致怎么做  
4. 书内或来源直接支持  
5. 在生活和工作里怎么用  
6. 边界  
7. 关系  
8. 相关概念 / 主题 / 来源

### 方法页禁止事项

- 只有结论，没有动作顺序
- 只有漂亮话，没有适用条件
- 迁移到生活和工作时不降强度

## 4. 概念页最低模板

### frontmatter 最低字段

- `type: concept`
- `status`
- `aliases`
- `related_concepts`
- `related_topics`
- `related_methods`
- `source_pages`

### 正文最低结构

1. 这个概念是什么意思  
2. 用大白话怎么讲  
3. 它在原资料里怎么出现  
4. 它有什么例子  
5. 它能拿来干什么  
6. 容易混淆的概念  
7. 强相关概念  
8. 关系  
9. 相关方法 / 主题 / 来源

### 概念页禁止事项

- 只有词典式定义
- 不解释它在原资料里如何出现
- 不给任何例子或用途

## 5. 案例页最低模板

### frontmatter 最低字段

- `type: case`
- `status`
- `case_type`
- `related_topics`
- `related_methods`
- `related_concepts`
- `source_pages`

### 正文最低结构

1. 这是什么案例  
2. 这个案例讲了什么  
3. 它体现了什么方法  
4. 它支撑了什么概念  
5. 它对应什么主题  
6. 它来自哪里  
7. 公式之外提醒了什么  
8. 关系  
9. 相关方法 / 概念 / 来源

### 案例页禁止事项

- 没有具体对象
- 没有具体场景
- 只有抽象结论

## 6. 领域页最低模板

当前可以先简化，作为导航页使用。

### 正文最低结构

1. 这个领域在处理什么长期问题  
2. 当前已有主题  
3. 当前已有方法  
4. 当前缺什么  
5. 后续待补的来源

当前证据不够时，允许先不建正式领域页。

## 七、什么时候建页，什么时候别建

### 建概念页的条件

- 它是稳定认识单元
- 后面还会复用
- 能解释、能举例、能连接

### 不建概念页的情况

- 只是章节标题
- 只是一次性判断
- 只是两个概念之间的关系句

### 建方法页的条件

- 有动作顺序
- 有适用条件
- 后面能复用

### 不建方法页的情况

- 只是态度
- 只是价值判断
- 只是一个结论句

### 建案例页的条件

- 有具体对象
- 有具体场景
- 有具体过程或典型路线

### 不建案例页的情况

- 只有定义
- 只有判断
- 只有抽象说明

### 建主题页的条件

- 多个知识单元都在往同一个问题聚拢
- 它是问题入口，不只是资料堆

### 暂不建正式页的情况

- 证据不够
- 当前名字还不稳定
- 只是阶段性迁移判断

这种情况先放 `outputs/`。

## 八、长文与短文的不同整理路径

### 短文

通常可以：

`来源页 -> 候选池 -> 概念 / 方法 / 案例 -> 索引`

### 长文 / 长书

必须多一层：

`长文整体导览 -> 章节工作层 -> 候选池 -> 来源页 -> 主题 / 方法 / 概念 / 案例 -> 索引`

### 多来源同主题资料

更适合：

`各自来源页 -> 交集候选池 -> 共用主题页 / 方法页 / 概念页`

## 九、写作口径规则

### 1. 先写来源支持什么

不要先写“我觉得它很像什么”。

### 2. 显式区分两层

至少要区分：

- 原文直接支持
- 当前整理后的保守迁移

### 3. 能降强度就降强度

不要动不动写：

- “这本书证明了……”
- “这套方法可以直接指导所有……”

更稳的写法是：

- “当前来源支持……”
- “当前可保守迁出……”
- “在生活和工作里可以低强度使用……”

### 4. 不要为了凑结构硬建页

页面不是越多越好，能稳定复用才值得建。

## 十、弱模型执行守则

如果是弱模型，建议把这些写死在提示词里。

### 弱模型硬规则

1. 先读总览、维护协议、README、相关索引  
2. 不修改 `raw/` 的知识内容  
3. 先建来源页  
4. 长文先走 `outputs/` 工作层  
5. 不把章节标题自动当概念  
6. 不把结论句自动当方法  
7. 没有具体对象、场景、过程的不当案例  
8. 每个正式页都要显式连到别的页  
9. 结构变化后更新总览和日志  
10. 迁移判断一律降强度

## 十一、可直接给弱模型的总提示词

```markdown
你现在不是在随意做笔记，而是在维护一座分层知识库。

你的目标是把资料整理成：
- 来源页
- 主题页
- 方法页
- 概念页
- 案例页

硬规则：
- 先读总览、维护协议、README 和相关索引
- raw 只保留原始资料，不把总结写回 raw
- 长书或长报告必须先建 outputs 工作层
- 来源页先于其他正式页
- 不要只做摘要，必须尝试抽概念 / 方法 / 案例 / 主题
- 概念页必须解释“是什么、怎么用大白话讲、在原资料里怎么出现、有什么例子、能拿来干什么”
- 方法页必须解释“是什么、适用于什么情况、大致怎么做、边界是什么”
- 案例页必须有具体对象、场景、过程
- 所有迁移判断都要降强度，明确不是原文原话
- 新页面至少要连到一个索引页和一个相关页面
- 改动影响结构时，更新总览和运行日志

默认输出顺序：
1. 来源页
2. 主题页
3. 方法页
4. 概念页 / 案例页
5. 索引与日志
```

## 十二、验收清单

一轮整理完成后，至少检查这些：

- 有没有来源页  
- 有没有只做摘要、没长知识单元  
- 有没有至少尝试抽主题 / 方法 / 概念 / 案例  
- 页面类型有没有用错  
- 方法页有没有适用条件  
- 概念页有没有例子和用途  
- 案例页是不是具体  
- 有没有把迁移判断说满  
- 新页面有没有显式链接  
- 总览和日志有没有更新  

如果这里有 4 条以上没过，这轮整理通常不算稳定。

## 十三、最容易翻车的地方

### 1. 把整理做成摘录仓库

修法：

- 强制要求每篇资料至少抽概念 / 方法 / 案例 / 连接中的几项

### 2. 只靠模型自由发挥

修法：

- 模板、顺序、规则、验收全部写死

### 3. 方法页和概念页边界混乱

修法：

- 能回答“怎么做”的优先放方法页
- 更像认识单元的放概念页

### 4. 主题页变成路标页

修法：

- 强制写“问题定义、当前结论、判断框架”

### 5. 所有东西都直接进 wiki

修法：

- 不稳的、还在迁移判断阶段的，先留在 `outputs/`

## 十四、最小可用版

如果模型很一般，也至少要做到：

1. 先读总览、维护协议和相关索引  
2. 建来源页  
3. 长文先建工作层  
4. 抽 1 个主题  
5. 抽 2 到 4 个最稳的方法 / 概念  
6. 补至少 1 个案例  
7. 更新总览与日志

做到这一步，通常已经比普通摘要式整理强很多。
