# 问题

怎样把一本长书、一篇长报告或一组长文，稳定拆成知识库真正能用的东西，而不是最后只得到一篇长摘要？

## 短答案

可复用的长文拆解，不该直接从“整本书 -> 总结页”跳过去。

更稳的路径是：

1. 先确认拆解目标和边界  
2. 先做章节级工作层  
3. 再从章节里抽候选概念 / 方法 / 案例 / 主题  
4. 先区分“原文直接支持”与“整理者迁移判断”  
5. 再决定哪些值得进正式知识页，哪些先留在 `outputs/`  
6. 最后做链接、去重和验收

一句话说，就是：

**先把长文拆成可处理的小单元，再把小单元整理成知识库单元。**

## 展开

## 一、这套方法想解决什么问题

长文拆解最容易翻车的地方有 4 个：

1. 只做摘要，不长知识单元  
2. 把所有章节标题都硬抽成概念  
3. 一上来就跨太大，把自己的理解说成原文结论  
4. 还没弄清整本书主线，就开始到处建页

所以这套方法的目标不是“压缩文字”，而是：

- 把长文拆成可复用的知识部件
- 让后续主题页、方法页、概念页、案例页能长出来
- 控制推理强度，不让迁移判断跑太远
- 让模型就算弱一点，也能按固定步骤做出七八成稳定结果

## 二、适用范围与不适用范围

### 适用范围

- 一本书
- 一篇长报告
- 一组同主题长文
- 一份章节结构清楚的长资料

### 最适合的场景

- 资料本身密度高，一次不可能直接整理完
- 用户希望最后沉淀进知识库，而不只是做读书笔记
- 希望后面能抽出方法、概念、案例，而不只是“这章讲了什么”

### 不适用范围

- 很短的新闻或短帖
- 只有结论、没有结构的碎片材料
- 时效性极强、但复用价值很低的内容

这类东西可以先做普通来源页，不必硬走“章节工作层”。

## 三、最终应该产出什么

一次合格的长文拆解，至少应产出这 4 层东西：

### 1. 工作层

这是中间层，不一定最后都进 `wiki/`，但必须有。

最低至少包括：

- 整体导览
- 章节级工作笔记或章节卡片
- 候选概念 / 方法 / 案例池

### 2. 来源层

至少要有一篇来源页，回答：

- 这份长文整体在讲什么
- 当前整理范围到哪里
- 已经稳定拆出了哪些知识单元
- 哪些结论还不能说死

### 3. 知识层

根据材料强度，长出下面这些正式页：

- 主题页
- 方法页
- 概念页
- 案例页

### 4. 结果层

把这次拆解总结成一篇方法提炼或阶段报告，留在 `outputs/`。

这个层常常用来放：

- 公式之外的作用
- 当前样板的迁移判断
- 还不适合直接回写到 `wiki` 的总结

## 四、标准流程

下面这套流程，是给弱模型也能执行的版本。

### 第 0 步：先确认“这次到底拆到哪”

不要默认整本书一次拆完。

先确定：

- 这次拆整本，还是只拆前两章 / 前五章
- 这次目标是“做样板”，还是“做完整整理”
- 这次更重心放在概念、方法，还是主题骨架

#### 本步输出

- 一句范围定义
- 一句这次拆解目标
- 一句当前不做什么

#### 例子

- “本次只拆《工程控制论》前两章，不提前把后续章节的结论写死。”

### 第 1 步：先读整份材料的骨架，不要先陷进细节

对长文，先读这些：

- 目录
- 前言 / 引言 / 摘要
- 各章标题
- 章节之间的顺序

目的是回答：

- 作者是按什么顺序组织问题的
- 整份材料的主线是什么
- 哪几章更像入口，哪几章更像展开

#### 本步输出

- 整体主线一句话
- 当前最值得优先拆的章节列表
- 初步候选主题 1 到 3 个

#### 禁止事项

- 还没看清结构，就开始大规模建概念页
- 只看关键词，不看章节顺序

### 第 2 步：建立章节级工作层

这是整套方法最关键的一步。

长文不要直接从整本跳到正式知识页，中间必须有“章节工作层”。

每章至少建立一张工作卡，哪怕只是 `outputs/` 中的一页。

### 第 3 步：每章固定回答 6 个问题

每章都按同一组问题来写，这样弱模型才不容易跑偏。

#### 章节工作卡固定 6 问

1. 这章在整本里的位置是什么  
2. 这章主要在解决什么问题  
3. 这章直接讲了哪些关键概念 / 方法 / 例子  
4. 这章最稳的结论是什么  
5. 这章有哪些东西可能能迁移，但现在还不能说太满  
6. 这章和前后章节怎么接

#### 每章最低产出

- 本章讲了什么
- 本章候选概念
- 本章候选方法
- 本章候选案例
- 本章的判断边界

### 第 4 步：从章节工作层抽“候选池”，先别急着建正式页

从所有章节工作卡里，先抽候选池：

- 候选概念池
- 候选方法池
- 候选案例池
- 候选主题池

这里的关键不是“抽得越多越好”，而是先收集，再去重。

#### 候选池阶段的原则

- 允许重复
- 允许不稳定
- 允许暂时名字很笨
- 先别急着正式命名

### 第 5 步：对候选池做第一次筛选

这个阶段才决定什么值得进正式知识库。

#### 概念页筛选标准

只有同时满足大部分条件，才建概念页：

- 它是稳定认识单元，不是一次性说法
- 它不只是某个章节标题
- 它后面还可能被复用
- 它能被解释、举例、连出去

#### 方法页筛选标准

只有同时满足大部分条件，才建方法页：

- 它不是纯结论，而是可重复执行的动作或判断顺序
- 它能回答“适用于什么情况”
- 它能回答“怎么做”
- 它不只服务一个狭窄场景

#### 案例页筛选标准

只有同时满足大部分条件，才建案例页：

- 有具体对象
- 有具体场景
- 有具体过程或典型进入路线
- 不是纯抽象定义或纯结论

#### 主题页筛选标准

只有同时满足大部分条件，才建主题页：

- 多个概念 / 方法 / 案例都在往同一个问题聚拢
- 它是一个真正的问题入口，不只是目录汇总
- 后面有继续扩展的空间

### 第 6 步：先分证据强度，再写正式页

这是防止跑飞的关键。

每个提炼出来的东西，都先判断属于哪一档：

#### A 档：原文直接定义或直接展开

- 可以明确写成“原书直接支持”

#### B 档：原文没直接下定义，但章节结构和内容强支持

- 可以写成“当前来源强支持”

#### C 档：在原文基础上的保守迁移

- 可以写，但必须明确说这是整理判断，不是原书原话

#### D 档：目前只是联想

- 暂时不要进正式页
- 可以先留在 `outputs/` 或待处理

### 第 7 步：正式写页时，顺序不要乱

更稳的顺序通常是：

1. 先写来源页  
2. 再立主题主线  
3. 再写方法页  
4. 再补概念页和案例页  
5. 最后回头统一链接和索引

原因很简单：

- 来源页决定这份材料整体怎么被理解
- 主题页决定这次整理到底围绕哪个问题展开
- 方法页和概念页如果先乱建，很容易变成散件

## 五、章节工作卡模板

下面这个模板，弱模型可以直接照抄。

### 章节工作卡最低模板

```markdown
# 第 X 章 - 标题

## 这章在整份材料里的位置

## 这章要解决什么问题

## 这章直接讲了什么
- 候选概念：
- 候选方法：
- 候选案例：

## 我整理后的最稳结论

## 当前可能能迁移出去的东西
- 只写低强度判断

## 当前边界
- 哪些不能说太满

## 和前后章节的关系
```

### 章节工作卡的硬规则

- 每章都要写“边界”
- 每章都要区分“直接讲了什么”和“我整理后的理解”
- 每章都至少抽一次候选概念 / 方法 / 案例

## 六、概念、方法、案例、主题的判定规则

这部分最好给弱模型写死。

### 什么情况下不要建概念页

- 只是章节标题
- 只是一次性判断
- 只是两个概念之间的关系句
- 只是用户当前随口的一种问法

### 什么情况下不要建方法页

- 只有结论，没有动作顺序
- 只适用于一个极小场景
- 其实只是“态度”或“口号”

### 什么情况下不要建案例页

- 只有抽象论证，没有具体对象
- 只有定义，没有场景
- 只有结论，没有过程

### 什么情况下不要建主题页

- 只是一个来源名
- 只是一个概念名
- 只有一个页面在支撑它
- 还看不出“问题入口”意义

## 七、写作口径规则

### 1. 先写材料支持什么，再写你觉得它像什么

顺序不能反。

### 2. 迁移判断必须降强度

不要写：

- “这本书就是人生方法论”
- “它可以直接指导一切工作和生活问题”

更稳的写法是：

- “当前可保守迁出一条处理复杂对象的骨架”
- “在生活和工作中可以低强度使用”

### 3. 不要把一章里所有小节都硬转成正式知识页

章节工作层可以多，正式知识页要克制。

## 八、弱模型执行提示词

下面这段可以直接给弱模型。

```markdown
你现在不是在写摘要，而是在做长文拆解。

你的目标不是压缩内容，而是从长文中稳定长出：
1. 来源页可用的整体主线
2. 章节级工作卡
3. 候选概念 / 方法 / 案例 / 主题
4. 原文直接支持与整理者迁移判断的分层

硬规则：
- 不要直接从整本书跳到总结页
- 每章都要写“这章解决什么问题”
- 每章都要抽候选概念 / 方法 / 案例
- 每章都要写“当前边界”
- 不要把章节标题自动当成概念页
- 不要把结论句自动当成方法页
- 没有具体对象、场景、过程的内容，不要当案例页
- 所有迁移判断都要降强度

输出顺序：
1. 整体主线
2. 章节工作卡
3. 候选池
4. 筛选后的正式知识单元建议
5. 当前不宜说死的地方
```

## 九、验收清单

一次长文拆解结束后，至少检查这些：

- 有没有章节级工作层  
- 有没有只做摘要、没长知识单元  
- 有没有把章节标题批量变成概念页  
- 有没有把自己的迁移判断写成原文原话  
- 有没有明确当前整理范围  
- 有没有写“当前不宜说死的结论”  
- 有没有至少长出来源页 + 一层正式知识页  
- 有没有留下下一步该继续拆哪部分

如果上面有 3 条以上没满足，这次拆解通常还不算稳。

## 十、最容易翻车的地方

### 1. 一上来就抽太大主题

修法：

- 先收范围，只拆前两章或前几章

### 2. 概念抽太碎

修法：

- 先放候选池，不急着建正式概念页

### 3. 方法抽成口号

修法：

- 强制补“适用于什么情况”和“怎么做”

### 4. 案例抽成定义

修法：

- 没有具体对象、场景、过程，就降回概念或来源说明

### 5. 把迁移理解说满

修法：

- 所有迁移句前先问一句：这是原文直接说的吗？

## 十一、最小可用版

如果模型比较弱，最小可用版只要求做到这些：

1. 先写整体主线  
2. 每章写工作卡  
3. 每章抽概念 / 方法 / 案例各至少 1 轮  
4. 建 1 个来源页  
5. 建 1 个主题页  
6. 建 2 到 4 个最稳的方法 / 概念页  
7. 把不稳的迁移判断先放进 `outputs/`

做到这一步，通常已经比“直接写一篇长摘要”强很多。
