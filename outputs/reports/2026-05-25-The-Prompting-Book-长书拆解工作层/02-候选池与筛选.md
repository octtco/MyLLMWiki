# The Prompting Book 候选池与筛选

## 主题候选

### 提示词设计

- 判断：创建正式主题页。
- 理由：这不是单个概念，也不是单个方法，而是围绕“怎样把 AI 交互从一次性问法推进到可控工作流”的局部问题空间。
- 支撑来源：`The Prompting Book`、[[GitHub - awesome-chatgpt-prompts]]、[[GitHub - awesome-ai-dev-prompts]]、[[Linux.do - 特殊风格焚决与 Prompt 思路]]、[[Linux.do - 生图的真神（焚决）]]。
- 边界：当前只整理工作定义和已有来源链，不把它写成完整 prompt engineering 学科页。

### Prompt 资产化

- 判断：暂留工作层。
- 理由：已有 [[AI 工具化知识工作流]] 足以承接；暂不需要独立主题页。
- 后续触发：如果后续继续积累 prompt 库、prompt marketplace、版本管理、评测和团队复用来源，再考虑升级。

## 方法候选

### 先拆 prompt 构件，再调模型输出

- 判断：暂留工作层。
- 理由：有方法潜力，但当前主要来自单书，需要后续在真实任务中验证。
- 最小动作雏形：先写角色与背景，再写任务和约束，再定输出格式，最后用示例和失败样本迭代。

### 先把输出格式变成契约，再接程序

- 判断：暂留工作层。
- 理由：结构化输出和 JSON/YAML 章节支持这条方法，但它更适合与现有工程来源交叉后再升。

### 先按边界输入测试 prompt，再谈优化

- 判断：暂留工作层。
- 理由：Handling Edge Cases 与 Prompt Optimization 共同支持，但还需要真实失败样本。

- 理由：名词太大，容易变成提示词设计主题的短版；当前可在主题页中解释。

### Prompt chaining

- 判断：暂不建概念页。
- 理由：可能更像方法或 workflow 技术点，当前只有单书支撑，先留候选。

### Context engineering

- 判断：暂不建概念页。
- 理由：已有多个来源提到上下文工程，但当前仓库还没有稳定定义；后续可结合 coding agent、RAG、项目记忆来源再开。

### Agents & Skills

- 判断：不建新概念页，回写 [[Skill]]。
- 理由：仓库已有 `Skill` 概念页，书中 `prompts -> skills -> agents` 的比喻适合补充边界，而不是新建复合概念。

## 案例候选

- 写作与内容：暂不建案例页，单书用例缺少具体对象和过程。
- 编程与开发：暂不建案例页，当前更适合补 [[AI coding framework]] 边界。

## Review Gate

- concept granularity：本轮没有把 `Prompt`、`Prompt chaining`、`Context engineering` 强行升概念页，避免薄页。
- organization pattern：新主题页采用“问题为什么会散 -> 主线 -> 递进展开 -> 边界”的链式写法。
- narrative smell：新主题页正文承担主要解释，页尾链接只做导航，不让链接替正文说话。
