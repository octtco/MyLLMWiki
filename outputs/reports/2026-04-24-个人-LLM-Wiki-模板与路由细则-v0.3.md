# 个人 LLM Wiki 模板与路由细则 v0.3

## 这份文档现在负责什么

[[2026-04-24-个人-LLM-Wiki-结构升级方案-v0.2]] 已经把方向讲清了：

- 现有结构不用推翻
- 但要补对象层和行动层
- 聊天、灵感、目标、待办不能再硬塞进 `主题 / 方法 / 概念`

这份 `v0.3` 不再继续讲“大方向对不对”，而是往下压一层，专门回答：

**知道该补什么以后，实际应该怎么落。**

它主要负责三件事：

1. 给出一套最小可执行的路由规则。
2. 给出聊天来源页、对象页、目标页、seed 页的初版模板。
3. 提前指出最容易出现的误判，避免仓库一开始就被抽象层吃掉。

## 最短版本：先记住这 8 句

1. 新材料先看“现在是什么状态”，不是先看“最后像什么页”。
2. 原文先落 `raw/`，不要还没保原件就急着提炼。
3. 还没熟的判断先落 `seeds/`，不要一上来就拔成方法论。
4. 想长期推进的事先进 `todo/goals/`。
5. 当前就要做的动作先进 `todo/tasks/`。
6. 人、项目、产品、长期对象先去 `wiki/entities/`。
7. 只有稳定到可长期复用的内容，才继续抽成 `topics / methods / concepts / cases`。
8. 只要一份材料的主要价值还是“保语境”，就先别急着抽象。

## 一、最小路由规则

以后新材料进来，先问四个问题。

### 第一步：它还是不是原始材料

如果它仍然主要是这些东西：

- 网页原文
- 书或文章正文
- 聊天记录原文
- 会议纪要原文
- 语音转写原文
- 原始截图、图片、附件

那就先去 `raw/`。

关键不是它“以后也许能抽成什么”，而是**先保住原件和语境**。

### 第二步：它是不是还没熟

如果它已经带一点你的判断，但现在还处于这些状态：

- 只是一个方向
- 只是一个感觉
- 只是一个还没验证的判断
- 还不知道最终该挂到哪个对象或目标下面

那就先去 `seeds/`。

`seeds/` 的作用，不是拖延整理，而是给未成熟内容一个合法落点。

### 第三步：它是不是行动控制信息

如果这条内容主要在回答这些问题：

- 为什么现在要做
- 接下来准备怎么推进
- 当前卡点是什么
- 下一步是谁来做、什么时候做

那它就不该先去 `wiki/`，而该先去 `todo/`。

更具体地说：

- 长期或中期方向 -> `todo/goals/`
- 当前要执行的动作 -> `todo/tasks/`

### 第四步：它是不是已经稳定到值得长期复用

只有在内容已经稳定到这种程度时，才继续进入 `wiki/`：

- 之后很可能被重复调用
- 不再依赖某一次具体语境才能成立
- 能明确回答它是哪一类正式页
- 放进正式知识层后不太容易立刻后悔

如果还没到这个程度，就先停在 `raw / seeds / todo / outputs` 其中一层。

## 二、四类常见材料的默认路线

## 2.1 聊天记录

默认路线：

```text
raw/chats/
-> wiki/sources/ 聊天来源页
-> 更新 entities / goals / tasks
-> 必要时再抽 topic / method / concept
```

聊天记录最容易犯的错，是一上来就把它拆成概念页。

这样会有两个问题：

1. 谁在什么语境下说的，会丢。
2. 原本只是一次局部判断，容易被误写成长期结论。

所以聊天记录首先要先保成来源，再决定它影响了哪个对象、哪个目标、哪条待办。

## 2.2 灵感

默认路线：

```text
seeds/inbox/
-> entity / goal / task
-> 变稳后再升 topic / method / concept / case
```

灵感最容易犯的错，是“刚记下来就想把它升成知识”。

更稳的做法是：

- 先记它为什么值得留
- 再记它可能挂到哪里
- 等它真的反复出现，或者已经和某个对象 / 目标稳定连上，再决定升页型

## 2.3 目标

默认路线：

```text
todo/goals/
-> 关联 entities / topics / methods / outputs
-> 拆成 tasks/
-> 阶段性结论先进 outputs/
-> 稳定部分再回写 wiki/
```

目标最容易犯的错，是被写成一个看起来很大的主题页。

但“目标”本来就不是在回答“现在知道什么”，而是在回答“现在为什么做、做到哪了、接下来怎么办”。

## 2.4 待办

默认路线：

```text
todo/tasks/
-> 做完后如有经验，写 outputs/
-> 经验足够稳，再抽 case / method
```

待办最容易犯的错，是把它包装成方法页，结果方法里写的其实只是当前这一轮要做什么。

方法页讲的是可复用动作，待办讲的是当前执行动作，这两件事不能混。

## 三、一个够用的决策表

| 材料类型 | 第一落点 | 第二步通常更新哪里 | 当前不要急着做什么 |
| --- | --- | --- | --- |
| 聊天记录 | `raw/chats/` | `wiki/sources/`，再看 `entities / goals / tasks` | 不要直接拆概念页 |
| 会议纪要 | `raw/chats/` 或后续单独的 `raw/meetings/` | 来源页 + 目标 / 待办 | 不要一上来写成主题总结 |
| 灵感碎片 | `seeds/inbox/` | `entity / goal / task` | 不要急着升方法页 |
| 长期目标 | `todo/goals/` | 关联对象页和知识页 | 不要硬写成主题页 |
| 具体待办 | `todo/tasks/` | 完成后进 `outputs/` | 不要伪装成正式知识页 |
| 阶段性分析 / 问答 | `outputs/reports/` | 稳定后再回写 `wiki/` | 不要一产出就升正式页 |
| 已经很稳的人 / 项目 / 产品对象 | `wiki/entities/` | 反向链接来源、目标、主题等 | 不要拿概念页替代对象页 |
| 已经稳定的抽象认识 | `wiki/topics/` / `wiki/methods/` / `wiki/concepts/` | 按页型继续长 | 不要越过来源和对象直接空写 |

## 四、模板：聊天来源页

聊天来源页的职责，不是把聊天重写成鸡汤摘要，而是：

> 先把这次对话在知识库里的位置钉住，再把稳定信息分发出去。

### 推荐 frontmatter

```yaml
---
type: source
source_type: chat
status: active
participants:
  - 某人
  - 某人
time_range:
  start: 2026-04-24
  end: 2026-04-24
privacy: private
quote_policy: restricted
raw_path: raw/chats/2026-04-24-某次对话/
updated: 2026-04-24
---
```

### 推荐正文模板

```markdown
# {{聊天标题}}

## 这次聊天在说什么

## 关键推进

## 稳定信息

## 暂时还不能抽象的部分

## 建议回写
- 更新哪些 entity
- 更新哪些 goal
- 新增哪些 task
- 是否值得抽 topic / method / concept

## 原始位置
- `raw/chats/...`
```
```

### 使用提醒

- 不要把整段聊天原文复制进来源页正文。
- 不要把一句还没坐实的话写成长期判断。
- 这页的重点不是“总结得多漂亮”，而是“后续知道该更新哪里”。

## 五、模板：对象页（`wiki/entities/`）

对象页负责承接“具体对象”，而不是“抽象词条”。

它最适合的对象有：

- 人
- 项目
- 产品
- 组织
- 账号
- 长期关注对象

### 推荐 frontmatter

```yaml
---
type: entity
entity_type: person | project | product | organization | place | collection
status: active
aliases: []
related_sources: []
related_goals: []
related_topics: []
updated: 2026-04-24
---
```

### 推荐正文模板

```markdown
# {{对象名}}

## 这是什么

## 为什么它现在重要

## 当前状态

## 关键关系

## 最近变化

## 当前未决问题

## 相关来源

## 相关目标

## 相关知识页
```
```

### 使用提醒

- 如果你写着写着更像在解释一个概念，那它可能不是对象页。
- 如果你写着写着全是任务推进，那它可能更像目标页。
- 对象页最重要的不是定义，而是“当前到底怎样”。

## 六、模板：目标页（`todo/goals/`）

目标页负责行动控制，不负责概念定义。

### 推荐 frontmatter

```yaml
---
type: goal
status: active
priority: high
horizon: short | medium | long
owner: liuyifu
related_entities: []
related_topics: []
related_methods: []
updated: 2026-04-24
---
```

### 推荐正文模板

```markdown
# {{目标名}}

## 目标一句话

## 为什么现在做

## 做到什么算完成

## 当前阶段

## 当前卡点

## 推进策略

## 相关对象

## 相关知识

## 下一步任务

## 最近更新记录
```
```

### 使用提醒

- 目标页要敢写阶段和卡点，不要只写宏大口号。
- 目标页不是主题页，它不负责系统梳理“围绕这个问题现在知道什么”。
- 目标页和任务页要分开：目标页回答方向，任务页回答动作。

## 七、模板：Seed 页

Seed 的任务很朴素：

> 先把东西接住，不让它在还没成熟时就被写坏。

### 推荐 frontmatter

```yaml
---
type: seed
status: inbox | incubating | routed | dropped
seed_type: idea | excerpt | hunch | observation | question
source_hint:
possible_destinations: []
updated: 2026-04-24
---
```

### 推荐正文模板

```markdown
# {{seed 名}}

## 原始内容

## 为什么值得留

## 现在的直觉

## 可能去向
- entity
- goal
- task
- topic
- method
- concept

## 当前不确定点
```
```

### 使用提醒

- Seed 不是失败页，也不是临时垃圾桶。
- 它的意义就在于允许“我先知道这东西值得留，但我现在还不急着定页型”。
- 只要后面已经明确去向，就别让它永远停在 inbox。

## 八、最容易出现的 6 种误判

### 1. 把聊天记录直接抽成概念

错因：只看抽象词，不看语境。

修正：先做聊天来源页，再决定更新哪些对象、目标或知识页。

### 2. 把灵感直接拔成方法

错因：看到一句有味道的话，就想立正式页。

修正：先放 `seeds/`，等它在不同场景里反复成立，再决定是不是方法。

### 3. 把目标写成主题

错因：把“我现在要达成什么”和“围绕某问题知道什么”混在一起。

修正：目标先去 `todo/goals/`，主题页只保留抽象组织工作。

### 4. 把待办写成方法页

错因：把本轮动作误当成可复用动作。

修正：本轮动作先留在 `todo/tasks/`，做完后再看能不能沉淀成方法。

### 5. 拿概念页替代对象页

错因：仓库里只有抽象页，没有承接“人 / 项目 / 产品”的地方。

修正：补 `wiki/entities/`，让具体对象回到具体对象页。

### 6. 还没稳就急着升正式页

错因：觉得“只有进 wiki 才算认真整理”。

修正：允许内容先停在 `raw / seeds / todo / outputs`，不要被正式页冲动带着跑。

## 九、建议的最小落地顺序

这版细则更适合按下面顺序落地，而不是一口气全改：

1. 先承认这四类材料的不同路线：聊天、灵感、目标、待办。
2. 先补 `entities` 与 `goals` 这两个新落点。
3. 先拿真实材料跑三次样板：
   - 一段聊天
   - 一个对象
   - 一个目标
4. 跑顺以后，再正式把模板写进仓库模板层。
5. 最后再决定要不要回头整理旧内容。

## 当前结论

`v0.2` 解决的是“方向是否成立”，`v0.3` 解决的是“落地时先怎么不写坏”。

如果后面真要把这套东西写进仓库规则，最值得先落的，不是更多抽象页，而是：

- 一条更稳的路由规则
- 一套聊天 / 对象 / 目标 / seed 模板
- 对“先别急着抽象”的共识

只有这层先稳住，后面的个人 LLM Wiki 才不会一开始就又退回“资料进来以后全被压成主题 / 方法 / 概念”的老问题。
