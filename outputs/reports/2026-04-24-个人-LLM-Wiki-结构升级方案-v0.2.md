# 个人 LLM Wiki 结构升级方案 v0.2

## 这份文档现在负责什么

这份文档不是在推翻当前知识库，而是在回答一个更具体的问题：

**如果最终目标不是“整理资料”，而是“你什么都可以丢进来，然后慢慢长成只属于你的个人知识库”，那现有结构还缺什么。**

当前判断是：

- 现有 `raw/`、`seeds/`、`wiki/`、`todo/`、`outputs/`、`_index/` 这套底盘是对的。
- 真正要补的，不是把 `主题 / 方法 / 概念` 推倒重来，而是给它们补上同级搭子。
- 也就是说，当前结构更像“资料知识库”，下一步要把它升级成“个人 LLM Wiki”。

这份方案稿主要负责三件事：

1. 说明现有结构和个人 LLM Wiki 目标是否冲突。
2. 说明聊天记录、灵感、目标、待办为什么不能直接硬塞进 `主题 / 方法 / 概念`。
3. 给出一条低风险升级路径，让仓库先长出对象层和行动层，再慢慢扩到更完整的个人知识库。

## 最短版本：先记住这 10 句

1. 现在的结构不冲突，但还不够完整。
2. `主题 / 方法 / 概念` 适合抽象知识，不适合直接承接所有个人材料。
3. 聊天记录首先是来源，不是概念页。
4. 灵感首先要被接住，不要一进库就拔高成方法论。
5. 目标属于行动控制层，不属于主题页。
6. 待办继续留在 `todo/`，不要假装成知识页。
7. 这套结构最缺的是对象层，也就是“人 / 项目 / 产品 / 长期对象”这一层。
8. 建议在 `wiki/` 下新增 `entities/`，在 `todo/` 下新增 `goals/`。
9. 以后要多问一句“这东西现在处于什么状态”，再决定它该去哪一层。
10. 升级顺序要保守：先补口径，再加目录，再跑样板，最后才考虑回头整理旧内容。

## 一、总判断：冲突不大，缺口很明确

当前这座库和个人 LLM Wiki 的目标并不冲突。

从结构角色看，它其实已经和那种思路很接近了：

- `raw/` 对应原始来源层。
- `wiki/` 对应被大模型持续编译出来的正式知识层。
- `outputs/` 对应工作层、问答层和回填候选层。
- `_index/` 对应索引、日志、规则和健康检查。
- `AGENTS.md`、[[维护协议]]、[[总览]]，已经在承担 schema 和控制面的作用。

真正的缺口主要有两类：

### 1. 现在更擅长整理“资料”，还不够擅长整理“你自己”

目前这套结构很适合：

- 网页
- 书
- 论文
- 仓库
- 长材料工作层

但如果后面要吃进：

- 聊天记录
- 会议纪要
- 灵感碎片
- 长期目标
- 项目推进过程
- 对人的持续判断

就会发现现有结构少了一层“对象页”和一层“行动页”。

### 2. 现在的主页型更偏认知轴，不够覆盖对象轴和行动轴

目前正式知识页的主轴是：

- `topics/`
- `methods/`
- `concepts/`
- `cases/`

它们适合回答的是：

- 围绕一个问题，现在知道什么
- 某个动作该在什么时候调用
- 一个概念最小是什么意思
- 一个抽象结构在具体场景里怎样发生

但它们不擅长直接回答这些问题：

- 这个人 / 项目 / 产品现在是什么状态
- 我现在正在推进什么
- 某段聊天到底改变了什么
- 这个灵感现在还只是线索，还是已经变成可执行目标

所以问题不是这几类页不对，而是**它们只覆盖了抽象知识层，还没有把对象层和行动层拉进来。**

## 二、为什么聊天记录、灵感、目标、待办不能直接塞进主题 / 方法 / 概念

### 1. 聊天记录

聊天记录天然带着这些信息：

- 谁说的
- 对谁说的
- 在什么时间说的
- 因为什么事情说的
- 当时推进到了哪一步
- 哪些地方只是临时判断，哪些地方已经变成稳定共识

所以聊天记录首先是**来源材料**，不是概念页。

如果一上来就直接抽象，很容易把“语境”抹掉，只留下几句失真的总结。

### 2. 灵感

大多数灵感的问题不是没价值，而是还太早。

它常常只有：

- 一个方向
- 一句判断
- 一个例子
- 一个还没成熟的感觉

这种东西如果立刻写成方法页，容易发空；写成主题页，容易发散；写成概念页，又常常太虚。

所以灵感最需要的不是“立刻归类”，而是“先被妥善接住”。

### 3. 待办

待办回答的是：

- 要不要做
- 谁来做
- 什么时候做
- 现在做到哪了

它本质上是行动控制信息，不是知识沉淀信息。

所以待办应该继续待在 `todo/`，而不是跑到 `wiki/` 冒充正式知识页。

### 4. 目标

目标比待办更高一层，但它也不是概念，也不是主题。

目标要处理的是：

- 为什么现在做
- 成功算什么
- 当前卡点是什么
- 下一步如何推进
- 依赖哪些对象和知识

这类东西更像“持续维护的行动对象”，而不是“围绕某个问题的认知组织页”。

## 三、建议的结构升级口径

在不推翻现有仓库的前提下，建议把结构理解成下面这套：

```text
raw/        原样来源
seeds/      未成熟但值得留的东西
wiki/       正式知识页
  sources/
  entities/   ← 建议新增
  topics/
  methods/
  concepts/
  cases/
  domains/
  relations/
todo/       行动层
  goals/      ← 建议新增
  tasks/
outputs/    工作层、问答、阶段产物、回填候选
_index/     协议、索引、日志、健康检查
assets/     图片、附件
```

这里最关键的，不是目录名本身，而是多了两层：

- **对象层**：`wiki/entities/`
- **行动层**：`todo/goals/` + `todo/tasks/`

## 四、页型方案

## 4.1 聊天来源页

建议路径：

```text
raw/chats/某次对话
-> wiki/sources/某次对话来源页
-> 再决定要不要更新 entities / goals / tasks / topics / methods / concepts
```

它回答的不是“这里出现了哪些概念”，而是：

> 这次对话发生了什么，推进了什么，哪些信息已经稳定到值得继续回写。

建议正文最少包含：

1. 这次聊天在说什么
2. 关键推进
3. 稳定信息
4. 暂时还不能抽象的部分
5. 建议回写位置

也就是说，聊天来源页首先是**保语境的中间站**。

## 4.2 对象页（`wiki/entities/`）

这是当前最值得补的一类正式页。

它回答的不是：

> 这个词是什么意思

而是：

> 这个人 / 项目 / 产品 / 对象现在是什么，为什么重要，最近发生了什么，和什么强相关。

适合进入对象页的东西包括：

- 人
- 项目
- 产品
- 组织
- 账号
- 长期关注对象
- 对你持续重要的具体事物

建议正文最少包含：

1. 当前这是什么
2. 为什么它现在重要
3. 当前状态
4. 关键关系
5. 最近变化
6. 当前未决问题
7. 相关来源
8. 相关目标 / 待办
9. 相关知识页

它和概念页的区别很简单：

- 概念页讲“最小定义”
- 对象页讲“当前状态”

## 4.3 目标页（`todo/goals/`）

目标页回答的是：

> 我现在为什么要做这件事、做到哪了、接下来怎么推进。

它不负责讲概念，也不负责组织主题，而是承担行动层的控制面。

建议正文最少包含：

1. 目标一句话
2. 为什么现在做
3. 做到什么算完成
4. 当前阶段
5. 当前卡点
6. 推进策略
7. 相关对象
8. 相关知识
9. 下一步任务
10. 最近更新记录

## 4.4 灵感 Seed

灵感不宜直接建正式页，更适合继续放在 `seeds/`。

它回答的是：

> 这个东西我知道还没成熟，但我先不丢。

建议一个 seed 最少保留：

1. 原始内容
2. 为什么值得留
3. 我现在的直觉
4. 可能去向
5. 当前不确定点

这样以后再分流时，就不会只剩一句失去来历的摘录。

## 五、分流规则：先问它现在是什么状态，再问它该长成什么页

以后新材料进来，不要先问“这该建什么页型”，而要先问：

**它现在处于什么状态。**

### 1. 原始材料

如果它仍然是：

- 网页原文
- 书的正文
- 聊天原文
- 会议记录原文
- 语音转写原文

那它首先应该进 `raw/`。

### 2. 未成熟判断

如果它已经带一点你的判断，但还不稳，还不知道最终该去哪，那就先去 `seeds/`。

### 3. 明确行动项

如果它的主要任务是推进，而不是沉淀，那就进 `todo/`：

- 目标进 `goals/`
- 待办进 `tasks/`

### 4. 本次任务产物

如果它是一次问答、分析、阶段方案、整理规程，先去 `outputs/`。

### 5. 稳定知识

只有当内容已经稳定到值得长期复用时，才进入 `wiki/`。

## 六、四类常见材料的默认流转方式

### 1. 聊天记录

```text
raw/chats/
-> wiki/sources/
-> 更新 entities / goals / tasks
-> 必要时再抽 topics / methods / concepts
```

原则：**先保语境，再抽知识。**

### 2. 灵感

```text
seeds/inbox/
-> entity / goal / task
-> 变稳后再升 topic / method / concept / case
```

原则：**先接住，不急着拔高。**

### 3. 待办

```text
todo/tasks/
-> 完成后如有经验，写 outputs/
-> 稳定经验再进 case / method
```

原则：**待办负责推进，wiki 负责沉淀。**

### 4. 目标

```text
todo/goals/
-> 链接相关 entities / topics / methods / outputs
-> 拆出具体 tasks/
-> 阶段性认知总结先进 outputs/
-> 真正稳定的部分再回写 wiki/
```

原则：**目标是控制页，不是定义页。**

## 七、抽象页什么时候才该出现

### 1. 什么时候建主题页

当你要回答的是：

> 围绕某个问题，当前库里已经知道什么。

它不是为了替代目标页，也不是为了替代对象页。

### 2. 什么时候建方法页

当一个动作已经在多个地方复用，而且真的值得单独回答：

> 它修什么卡点，这一步到底在做什么，最小怎么做，什么时候该调用。

### 3. 什么时候建概念页

当某个词反复被调用，而且值得被单独定义清楚。

### 4. 什么时候先别建

如果它还只是：

- 一次聊天里的临时判断
- 某个目标下的阶段策略
- 某个项目独有的做法
- 还没稳定的灵感

那就先不要急着升成正式知识页。

## 八、低风险迁移顺序

升级顺序建议很保守。

### Phase 1：先补口径，不动旧内容

先把这些规则在方案和协议里说清：

- 聊天去哪里
- 灵感去哪里
- 目标去哪里
- 对象页负责什么

### Phase 2：只加最小目录

先只考虑补这几个：

- `raw/chats/`
- `wiki/entities/`
- `todo/goals/`

不要一口气加很多新页型。

### Phase 3：只跑三个样板

拿真实材料先试三条链：

1. 一段聊天记录
2. 一个长期目标
3. 一个具体对象

先验证这条链能不能跑顺：

```text
raw -> source -> entity / goal / task -> 必要时再抽 wiki 抽象页
```

### Phase 4：样板跑顺后再补模板

再正式补：

- 聊天来源页模板
- 对象页模板
- 目标页模板
- seed 模板

### Phase 5：最后再考虑回头整理旧内容

等新链条稳定以后，再看这些问题：

- 有没有旧的主题页其实更像对象页
- 有没有旧的 outputs 更适合回写到 goal / entity
- 有没有零散 seed 可以顺手分流

## 九、这版升级解决的是什么问题

### 1. 不再逼所有材料一上来都抽象化

不是每份材料都得立刻变成主题、方法、概念。

### 2. Agent 的入口会更清楚

以后问“我最近该推进什么”，Agent 会先看 `todo/goals/` 和 `todo/tasks/`；
问“这个项目最近发生了什么”，它会先看 `wiki/entities/`；
问“这套做法背后的方法是什么”，它才会回到 `wiki/methods/`。

### 3. 仓库会更像“你的库”，而不只是“资料库”

书、网页、聊天、灵感、目标、待办终于可以放进同一个系统里，但又不会互相挤坏。

## 十、当前建议的下一步

当前最合适的下一步，不是直接大改仓库，而是继续把这套方案压成更可执行的一层。

建议后续补一版：

**《个人 LLM Wiki 模板与路由细则 v0.3》**

重点只做这些事情：

1. 聊天来源页模板
2. 对象页模板
3. 目标页模板
4. seed 模板
5. 一张“新材料进来后怎么分流”的决策表

这样做的好处是：

- 先把“以后怎么长”定清
- 再开始动目录和模板
- 中间不需要大规模返工现有内容

## 当前结论

这座知识库要升级成个人 LLM Wiki，不需要换仓库，也不需要推翻当前架构。

更稳的做法是：

- 保留现在的 `raw / seeds / wiki / todo / outputs / _index`
- 让 `主题 / 方法 / 概念` 退回抽象知识层
- 补出 `entities` 作为对象层
- 补出 `goals` 作为行动层
- 让聊天、灵感、目标、待办各自回到更适合自己的落点

这样它才会从“可整理资料的库”，慢慢长成“能持续编译你的材料、判断、对象和行动的个人知识系统”。
