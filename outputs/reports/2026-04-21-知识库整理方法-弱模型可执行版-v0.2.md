# 知识库整理方法（弱模型可执行版）v0.2

## 这份文档现在负责什么

这份文档现在不再只是“给弱模型的一张操作小抄”，而是一份仓库级整理总规程。

它主要负责这几件事：

- 这座知识库最小该怎样理解
- 不同类型资料该走哪条整理路径
- 正式页大致按什么顺序长出来
- 来源页、主题页、方法页、概念页、案例页各自负责什么
- 写作口径怎么收紧，才不容易越写越飘
- 弱模型进入任务后，哪些动作必须写死

但它不再单独包办“长书、长报告、长文合集到底怎么一步一步拆”。

这一块现在交给：[[2026-04-21-长文拆解的可复用方法-v0.2]]。

更准确地说：

- **这份文档负责总调度**
- **[[2026-04-21-长文拆解的可复用方法-v0.2]] 负责长材料专项分支**

所以这份 `v0.2` 的重点，不是把另一份文档整篇重讲一遍，而是把两者的分工和接口钉住。

## 最短版本：先记住这 10 句

1. 先认任务类型，不同资料走不同流程。
2. 先读总览、维护协议、README 和相关索引，不要一上来就写页。
3. `raw/` 只放原始材料，不把知识性总结写回去。
4. 正式整理一般先有来源页，再往下长别的页。
5. 长材料不要直接冲进 `wiki/`，先走 `outputs/` 工作层。
6. 候选池先收集、再筛页型，不要边读边乱建正式页。
7. 主题、方法、概念、案例各管各的，不要偷彼此的活。
8. 正文优先，链接嵌在正文里，不要让链接替正文解释。
9. 先写来源支持什么，再写你整理出了什么；所有迁移判断都降强度。
10. 只要结构或可发现性变了，就更新索引和日志。

## 一、这座知识库最小怎么理解

整理前，先把这座库理解成 4 层，而不是一堆散文件。

### 1. 原始层 `raw/`

这里放原始资料、来源包、附件、图片、本地化保存的原件。

这层的任务是：

- 保留来源
- 保留证据
- 保留原始结构

它**不负责**：

- 正式知识总结
- 主题判断
- 方法迁移
- 关系层组织

一句话说：`raw/` 是来源保存层，不是知识改写层。

### 2. 编译层 `wiki/`

这里放正式知识页。

当前最小页型包括：

- `wiki/sources/`
- `wiki/topics/`
- `wiki/methods/`
- `wiki/concepts/`
- `wiki/cases/`
- `wiki/domains/`
- `wiki/relations/`

但不是每轮都必须把这些页型全建齐。

### 3. 结果层 `outputs/`

这里放阶段性产物，而不是默认次级垃圾桶。

它适合放：

- 章节工作层
- 样板整理
- 中间报告
- 还不够稳的迁移判断
- 问答输出
- 暂时不该直接回写到 `wiki/` 的内容

尤其是长书、长报告、高密度资料，通常都要先在这里过一层。

### 4. 控制面 `_index/`

这里管的是运行面：

- 总览
- 索引
- 待处理
- 日志
- 健康检查
- 维护协议

如果这层不跟着更新，知识页就算写出来了，也会越来越难找、越来越难维护。

## 二、任务开始前固定先读什么

不管是不是弱模型，进任务后都不要马上发挥。

至少先读：

1. `_index/总览.md`
2. `_index/维护协议.md`
3. `raw/README.md`
4. `wiki/README.md`
5. `outputs/README.md`
6. 与当前任务最相关的索引页

如果这次任务不是从零开始，而是在已有样板上继续整理，再补读：

- 当前来源页
- 当前最相关的主题页
- 当前最相关的方法页
- 当前最相关的概念页
- 最近一次对齐这套口径的输出页

这一步的目的不是“多读点背景”，而是先把当前仓库的组织口径对齐，不让模型每次都自己发明一套结构。

## 三、先判断任务类型，再选流程

整理资料前，先判断这次是哪一类任务。

### 1. 短资料

比如：

- 一篇结构不太长的文章
- 一篇帖子
- 一个 README
- 一份边界清楚、信息量不算爆炸的来源

这类资料通常可以走：

`来源页 -> 候选池 -> 正式页 -> 索引与日志`

一般不需要很重的工作层。

### 2. 长资料

比如：

- 一本书
- 一篇长报告
- 一组章节结构清楚的长文
- 一份高密度、不能一次稳定总结的长材料

这类资料**不要直接从整本材料跳到正式页**。

要先走：

`范围定义 -> 骨架阅读 -> outputs 工作层 -> 候选池 -> 页型筛选 -> 正式页`

这条分支的细节，统一按：[[2026-04-21-长文拆解的可复用方法-v0.2]] 来跑。

### 3. 多来源同主题资料

比如：

- 多篇文章都在讨论同一个问题
- 一个 repo、一篇帖子和一篇评论在讲同一个对象
- 多个来源共同支持一个主题或一个方法

这类资料更稳的路径通常是：

`各自来源页 -> 交集候选池 -> 共用主题 / 方法 / 概念 -> 索引与日志`

不要把多来源同主题任务误做成“先写一个大主题页，再回头补来源”。

### 4. 只是做阶段性输出

如果这次任务的目标只是：

- 回答问题
- 出一份报告
- 做一页样板
- 先放一份还不稳定的整理判断

那就先落到 `outputs/`。

`outputs/` 不是失败区，而是缓冲层。

## 四、默认整理主路径

这座库当前更稳的整理主路径，不是某条“唯一阅读链”，而是一条施工顺序。

### 第 0 步：先定范围、目标和当前不做什么

至少先写清：

- 这次在整理什么
- 这次只整理到哪
- 这次更想先得到什么
- 这次明确不做什么

这一步不做，后面最容易边整理边扩张。

### 第 1 步：先建或更新来源页

来源页通常是正式整理的入口。

因为后面无论是概念、方法、主题还是案例，都需要一个能回到来源的支点。

所以大多数情况下：

- 先有来源页
- 再往下长别的正式页

### 第 2 步：判断要不要先建工作层

只要材料够长、够密、够容易写散，就别直接冲进 `wiki/`。

更稳的做法是先在 `outputs/` 建工作层，比如：

- 章节工作卡
- 当前主线
- 候选池
- 当前边界

如果这一步触发了，就按 [[2026-04-21-长文拆解的可复用方法-v0.2]] 往下跑。

### 第 3 步：抽候选池

候选池至少包括：

- 候选主题
- 候选方法
- 候选概念
- 候选案例

这一步先允许重复、撞名、名字不优雅、之后被降级。

重点是先收，不是边收边定死。

### 第 4 步：筛页型

不要把候选池直接全建成正式页。

先判断：

- 这更像主题，还是更像概念
- 这更像方法，还是更像一句结论
- 这更像案例，还是只是例子
- 这页值得正式沉淀，还是只该先留在 `outputs/`

### 第 5 步：按顺序落正式页

更稳的默认顺序通常是：

1. 来源页
2. 主题页
3. 方法页
4. 概念页
5. 案例页
6. 索引与日志

这不是铁律，但对于弱模型来说，这个顺序能明显减少乱长。

### 第 6 步：补显式链接

每个正式页至少要满足：

- 能从一个索引页进入
- 能连到一个上游页或来源页
- 能连到一个平级页或下游页

不要留下“正文写完了，但图谱里进不去也出不来”的孤页。

### 第 7 步：更新控制面

只要结构、可发现性或运行状态变了，就更新：

- `_index/总览.md`
- 对应索引页
- `_index/待处理.md`（如果有遗留问题）
- `_index/log.md`

## 五、页面分工规则

这部分现在要和长文拆解规程对齐，不再各说各话。

### 1. 来源页负责总入口和证据收口

来源页最重要的不是“总结得多漂亮”，而是：

- 这份来源现在到底在支持什么
- 当前整理范围收到了哪里
- 后面已经从它拆出了什么
- 哪些判断还不能说太满

来源页不该承担：

- 包办整套主题解释
- 包办所有方法说明
- 把正式知识全塞进自己正文里

### 2. 主题页负责局部组织

主题页回答的是：

为了讲清一个问题，这一组概念、方法、案例在这里该怎么排、怎么串。

重点是：**在这里**。

所以主题页更像一个问题入口下的组织页，而不是全库唯一正统路径。

主题页不该写成：

- 相关页面总表
- 单纯路标页
- 换壳版概念页

### 3. 方法页负责调用时机

方法页现在不再主打“步骤大全”，而是主打：

- 卡点是什么
- 为什么这里最容易做错
- 这一步真正要补什么
- 到什么时候它该被拿出来用

也就是说，方法页的重点是“什么时候调用、为什么在这里调用”，而不是“把所有相关概念再讲一遍”。

### 4. 概念页负责最小公共定义

概念页的工作，是把一个词在当前知识库里先钉住。

更稳的方向是：

- 这个概念最小指什么
- 它常和哪些问题接上
- 这页不负责什么
- 哪些主题或方法会重点调用它

概念页不该偷偷长成：

- 某个主题的总组织链
- 某条方法的完整解释
- 一张大而全的导航页

### 5. 案例页负责具体落点

案例页不是抽象观点的装饰品。

它至少要保住：

- 具体对象
- 具体场景
- 具体过程或进入路线

如果没有这些，很多时候还不如先把例子留在来源页、方法页或主题页里。

### 6. 领域页和关系页不是每轮必建

这两类页当前仍然有用，但不是每次整理都必须出。

更稳的做法是：

- 领域页在导航价值已经明显时再建
- 关系页在关系本身已经成熟到值得单独解释时再建

不要为了让页型齐全，硬把它们补出来。

## 六、每类页面的最低骨架

这里不再保留一套和长文拆解规程平行的旧模板，而是统一成当前口径。

### 1. 来源页最低骨架

至少回答：

1. 这份来源当前在支持什么
2. 这次整理范围收到哪里
3. 为什么它值得继续提炼
4. 当前哪些结论不能说太满
5. 已经拆出了哪些主题 / 方法 / 概念 / 案例
6. 相关来源路径和回链在哪里

### 2. 主题页最低骨架

至少回答：

1. 这页到底在回答什么问题
2. 为什么这个问题容易被做散
3. 当前最稳的主线是什么
4. 这条主线怎样一段一段展开
5. 当前边界在哪里

### 3. 方法页最低骨架

至少回答：

1. 这页在解决什么卡点
2. 为什么这里最容易做错
3. 这一步真正要补什么
4. 到什么时候该调用它
5. 在不同场景里会怎么变形
6. 边界在哪里

### 4. 概念页最低骨架

至少回答：

1. 这个概念最小指什么
2. 它常和哪些问题接上
3. 这页不负责什么
4. 哪些主题或方法会重点调用它
5. 边界在哪里

### 5. 案例页最低骨架

至少回答：

1. 这是什么具体对象
2. 它发生在什么场景里
3. 过程或进入路线是什么
4. 这个案例帮助说明了什么
5. 边界在哪里

## 七、写作口径规则

### 1. 先写来源支持什么，再写你整理出了什么

顺序不能反。

更稳的写法是：

- 先交代原文直接说了什么、支持了什么、展示了什么
- 再交代你在这个基础上整理出的低强度判断是什么

不要一上来就写“我觉得它本质上是什么”。

### 2. 正文优先，链接嵌在正文里

正式页首先要自己成立。

所以：

- 解释优先进正文
- 链接优先放在正文第一次真正需要它的时候
- 文末的“相关概念 / 相关方法 / 相关主题”只能当附录，不能承担主要解释任务

### 3. 所有迁移判断都降强度

不要轻易写：

- “这本书证明了……”
- “这套方法可以直接指导一切……”

更稳的写法是：

- “当前来源支持……”
- “当前可保守迁出……”
- “在生活和工作里可以低强度使用……”

### 4. 不要为了凑结构硬建页

不是页越多越好。

如果一个东西现在：

- 名字还不稳
- 证据还薄
- 只是阶段性迁移判断
- 还明显更像工作层产物

那就先留在 `outputs/`，别急着回写到 `wiki/`。

## 八、给弱模型的执行规则

如果这份规程要给较弱模型直接执行，建议把这些动作写死。

### 弱模型硬规则

1. 先读总览、维护协议、README 和相关索引。
2. 不把知识性总结写回 `raw/`。
3. 正式整理一般先建或更新来源页。
4. 长材料先走 `outputs/` 工作层，不直接从整本材料跳正式页。
5. 看到章节标题，不要自动当概念页。
6. 看到一句判断，不要自动当方法页。
7. 没有具体对象、场景、过程，不要硬建案例页。
8. 新正式页至少要显式连到别的页和索引页。
9. 结构变化后，更新总览、索引、待处理和日志。
10. 所有迁移判断一律降强度。

### 遇到长材料时的追加规则

如果对象是长书、长报告或长文合集，再额外加上：

1. 先收范围，不默认一次拆完整体。
2. 先看骨架，不先陷进细节。
3. 先建章节工作层。
4. 先收候选池，再筛正式页。
5. 页型分工和成文规则按 [[2026-04-21-长文拆解的可复用方法-v0.2]] 执行。

## 十、验收清单

一轮整理完成后，至少检查这些：

- 有没有明确当前整理范围
- 有没有来源页或来源页更新
- 如果是长材料，有没有先建工作层
- 有没有先抽候选池，再筛正式页
- 页型有没有用错
- 主题页有没有真的在回答一个问题
- 方法页有没有在讲调用时机，而不是只讲概念或口号
- 概念页有没有守住最小公共定义，没有偷偷长成主题页
- 案例页是不是保住了具体对象、场景、过程
- 正文不点链接时，能不能继续顺着读
- 迁移判断有没有说满
- 新页面有没有显式链接
- 总览、索引、待处理和日志有没有跟着更新

如果上面有 4 条以上过不了，这轮整理通常还不算稳。

---

## 十一、最容易翻车的地方

### 1. 把整理做成摘录仓库

修法：

- 不要停在摘要
- 至少尝试抽主题 / 方法 / 概念 / 案例中的若干

### 2. 所有东西都直接进 `wiki/`

修法：

- 不稳的、还在迁移判断阶段的，先放 `outputs/`
- 长材料先有工作层，再谈正式页

### 3. 来源页跳过不做

修法：

- 先补来源页，再让别的正式页往它回链

### 4. 方法页和概念页边界混乱

修法：

- 更像“什么时候该调用”的，优先放方法页
- 更像“这个词最小指什么”的，优先放概念页

### 5. 主题页写成导航页

修法：

- 强制先把正文写成连续回答
- 文末栏目降级成附录

### 6. 链接和正文分家

修法：

- 把关键概念尽量前置到正文第一次真正需要它的时候
- 不要把解释主要藏到文末

### 7. 把迁移判断说得太满

修法：

- 每写一句更强判断前，先问一句：这是来源直接支持的吗？

## 十二、这份总规程和长文拆解规程怎么配合

现在两者的关系，最好这样理解：

### 这份《知识库整理方法》负责：

- 仓库分层怎么理解
- 任务先怎么分流
- 默认整理主路径是什么
- 页型各自负责什么
- 写作口径怎样收紧
- 弱模型有哪些硬规则
- 验收和翻车修法怎么做

### 《[[2026-04-21-长文拆解的可复用方法-v0.2]]》负责：

- 长材料的范围收束
- 骨架阅读
- 章节工作层
- 候选池
- 页型筛选
- 长材料正式页怎么写，才不容易写散

### 两者怎么接

更稳的顺序是：

1. 先用这份总规程判断任务类型
2. 如果是长材料，就切到 [[2026-04-21-长文拆解的可复用方法-v0.2]] 跑专项流程
3. 长材料专项流程跑完后，再回到这份总规程里继续处理链接、索引、日志、回写边界这些仓库级动作

也就是说：

- **这份文档负责总调度**
- **长文拆解文档负责长材料细部施工**

这样以后继续加规则时，也不容易两边口径漂开。

## 十三、最小可用版

如果模型能力一般，至少也要做到：

1. 先读总览、维护协议和相关索引
2. 建或更新来源页
3. 长材料先建工作层
4. 抽 1 个主题
5. 抽 2 到 4 个最稳的方法 / 概念
6. 补至少 1 个案例（如果证据足够）
7. 更新总览与日志

做到这一步，通常已经比“只写一篇摘要”稳定很多。
