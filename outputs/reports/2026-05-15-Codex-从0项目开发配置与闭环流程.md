# Codex 从 0 项目开发配置与闭环流程

## 问题

我是程序员，过去主要基于已有项目新增功能或修 bug，所以裸用 Codex 通常够用。现在要从 0 开发一个完整项目，只有大致想法，担心产品、架构、UX、QA、上线风险这些非纯编码部分把控不好。应该怎么配置 Codex、skill、MCP 和执行流程？

## 短答案

这套流程可行，但不要把 Codex 配成一个长期常驻的“虚拟公司”。

更稳的做法是：

```text
Codex 主执行器
+ 项目 AGENTS.md
+ docs/ 规格层
+ 少量高价值 skill
+ 少量 MCP
+ 产品 / 架构 / UX / QA 关卡
+ 小切片反馈闭环
```

核心不是让多个角色一直发言，而是把每个角色的判断压成可执行产物：

- 产品经理视角产出 `product.md`
- 架构师视角产出 `architecture.md` 和 `decisions.md`
- UX 视角产出用户流程、页面状态和视觉约束
- QA 视角产出 `qa.md`、测试清单和验收结果
- Codex 负责在这些约束下实现、验证、修正

一句话：

**角色是视角，skill 要产出工件；流程要形成闭环，不要停在人格扮演。**

## 一、工程控制论判断

根据知识库中《工程控制论（第三版）》和[[控制论式系统思维]]的整理，这套流程可以用控制论骨架判断。

| 控制论问题 | 从 0 项目里的对应物 | 结论 |
| --- | --- | --- |
| 对象边界 | `product.md` 里写清做什么、不做什么、MVP 到哪 | 必须有，不然范围会发散 |
| 系统模型 | `architecture.md` 里的技术栈、模块边界、数据流 | 必须有，不然 Codex 会在局部文件里盲改 |
| 状态观测 | `tasks.md`、测试、浏览器验证、运行日志、用户反馈 | 必须有，不然看不见项目真实状态 |
| 反馈回路 | 每个小功能后 review、test、build、更新文档 | 核心，不做就变开环 |
| 稳定性 | 先 MVP 可跑、可测、可部署，再谈高级功能 | 正确顺序 |
| 质量指标 | `qa.md` 的验收标准、性能底线、体验底线 | 必须写，不然“做好了”没有尺子 |
| 约束进入设计 | 时间、预算、平台、技术栈、能力边界、安全要求 | 要提前写，不然后面返工 |
| 解耦协调 | 产品 / 架构 / UX / QA 只在关卡调用 | 对，不要角色常驻制造噪声 |
| 不确定性 | 风险清单、假设验证、用户反馈、线上错误 | 要纳入，不要假装需求一开始就准 |

所以这套方案成立的原因是：它不是让 Codex 一次性生成完整项目，而是把项目变成可观测、可控制、可修正的小闭环。

```text
想法 -> 规格 -> 架构 -> 小切片实现 -> 验收 -> 反馈修正 -> 下一切片
```

最大风险是反馈延迟。一次做太大，等到最后才验收，偏差会被放大。

建议每轮功能切片控制在 0.5-2 天能闭环。

## 二、第一阶段 Codex 配置

先不要装太多。

第一阶段目标是从模糊想法推进到 MVP，不是搭建全自动研发平台。

### 2.1 Codex 基础设置

建议保持：

```text
model: 使用当前 Codex 默认强模型
reasoning_effort: medium
sandbox_mode: workspace-write
approval_policy: on-request
```

复杂架构设计、疑难 bug、大范围重构时，再临时提高 reasoning effort。

### 2.2 必备 AGENTS.md

每个新项目根目录放 `AGENTS.md`。它是项目约束层，不是闲聊提示词。

建议内容包括：

- 项目目标
- 技术栈
- 目录约定
- 代码风格
- 测试命令
- 提交规则
- 安全边界
- 哪些操作必须先确认
- Done 标准

最小模板：

```md
# Project Agent Guide

## 回答与执行

- 默认使用中文回答。
- 任何代码修改前，先说明方案、改动点、风险和验收方式，等我确认后再动手。
- 如果任务涉及 OpenAI / Codex / API / Apps SDK，必须先查 OpenAI developer docs MCP。
- 如果任务涉及新框架、新库、新 API，优先查 Context7 或官方文档。
- 如果任务涉及前端 UI，完成后必须启动本地预览并用浏览器验证。

## 项目推进方式

从 0 开发时，先维护这些文件：

- `docs/product.md`：产品目标、用户、MVP 边界
- `docs/architecture.md`：技术栈、模块边界、数据流
- `docs/decisions.md`：关键决策记录
- `docs/tasks.md`：当前任务拆分
- `docs/qa.md`：验收标准、测试路径、发布检查

大功能流程：

1. 先澄清需求和边界。
2. 写或更新 docs。
3. 拆成可独立完成的小任务。
4. 一次只实现一个任务。
5. 实现后跑 lint / typecheck / test / build。
6. 前端改动必须视觉检查。
7. 总结改了什么、怎么验证、剩余风险。

## 工程默认标准

- 不引入新生产依赖，除非先说明理由并获得确认。
- 不做无关重构。
- 不改动用户已有改动。
- 不提交密钥、token、cookie、账号信息。
- `.env*` 只写示例，不写真实值。
- 优先保持实现简单，避免过早抽象。
- 可读性优先于炫技。
```

## 三、第一批 Skills

第一批只装能补短板的 skill。

### 3.1 推荐安装

```bash
npx skills add https://github.com/mattpocock/skills --skill grill-me
npx skills add https://github.com/jwynia/agent-skills --skill requirements-analysis
npx skills add https://github.com/jwynia/agent-skills --skill system-design
npx skills add sleekdotdesign/agent-skills@sleek-design-mobile-apps
npx skills add https://github.com/obra/superpowers --skill test-driven-development
npx skills add https://github.com/obra/superpowers --skill verification-before-completion
```

安装前可以用 `npx skills find <keyword>` 或 skills.sh 复核当前包名和维护状态。

如果项目后面确定是 iOS 原生或 Flutter，再补平台专项 skill；在产品边界没定之前，不先把技术栈锁死。

### 3.2 角色对应关系

| 角色视角 | 推荐 skill | 主要用途 |
| --- | --- | --- |
| 产品经理 | `grill-me` + `requirements-analysis` | 把模糊想法翻译成真实问题、用户、MVP、做 / 不做和验收标准 |
| 架构师 | `system-design` | 从明确需求推技术架构、模块边界、walking skeleton |
| UI / UX | `sleek-design-mobile-apps` | 做移动 App 界面、主流程、页面状态和视觉一致性 |
| QA | `test-driven-development` + `verification-before-completion` | 先想测试，做完不靠感觉收工 |
| Reviewer | Codex 内建 review 先够用 | PR 前查风险、缺测试、逻辑漏洞 |

### 3.3 第二批按需装

项目进入真实复杂度后，再考虑：

```bash
npx skills add https://github.com/jwynia/agent-skills --skill architecture-decision
npx skills add https://github.com/mattpocock/skills --skill improve-codebase-architecture
npx skills add https://github.com/softaworks/agent-toolkit --skill qa-test-planner
npx skills add https://github.com/jwynia/agent-skills --skill code-review
npx skills add https://github.com/openai/skills --skill sentry
```

第二批不要一开始装满。项目没到那个阶段，装了也只是噪声。

## 四、第一批 MCP

MCP 只接 Codex 原本够不到的东西。

第一批建议：

| MCP | 用途 | 是否第一批 |
| --- | --- | --- |
| `openaiDeveloperDocs` | 查 OpenAI / Codex / API 官方文档 | 是 |
| `context7` | 查新框架、新库、新 API 文档 | 是 |
| `pencil` | App / 界面设计稿与视觉协作 | 已有 |

示例配置：

```toml
[mcp_servers.openaiDeveloperDocs]
url = "https://developers.openai.com/mcp"

[mcp_servers.context7]
command = "npx"
args = ["-y", "@upstash/context7-mcp"]
```

按需再加：

| MCP | 什么时候加 |
| --- | --- |
| Playwright | 做 Web、H5、管理后台或 App 内 WebView 时 |
| Figma | 真有设计稿要还原时 |
| GitHub | 开始做 issue、PR、release 时 |
| Sentry | 项目上线并接错误监控后 |
| Database MCP | 数据模型稳定、确实需要查库后 |
| Slack / Linear / Notion | 协作复杂后 |

不要一开始接一堆 MCP。MCP 是能力入口，也是风险入口。

## 五、为什么不建议装“虚拟产品经理”大角色包

大角色包容易变成人格扮演，而不是流程产物。

坏形态是：

```text
我将以资深产品经理视角帮你分析……
```

然后输出很多像样的话，但没有把项目压成硬东西：

- 用户是谁
- 痛点是否真实
- MVP 边界
- 不做什么
- 验收标准
- 风险假设
- 下一步任务

从 0 做项目，最怕的不是没人扮演产品经理，而是没有把模糊想法变成可执行规格。

新补充的[[Linux.do - 产品经理、客户经理以及技术头儿有什么区别]]把这件事说得更准：产品经理不只是接需求、画原型、催研发，而是要判断方向和取舍，把客户嘴里说的功能翻译成背后的真实问题，再决定这件事要不要进入产品长期能力。

所以更稳的方式是：

```text
不要装“产品经理人格”
要装“需求澄清流程”
```

对应关系：

- `grill-me`：追问，把模糊点逼出来
- `requirements-analysis`：整理需求、用户故事、验收标准
- `system-design`：把需求转成架构和工程边界

产品关卡真正要守住的是：

- 先问真实问题，不直接搬运客户原话。
- 先判断做 / 不做 / 以后再做，不把所有请求塞进 MVP。
- 先提炼通用产品能力，不为了单个客户把产品做成定制工坊。
- 先把会影响技术选型的产品约束讲清楚，例如是否强依赖 iOS 原生能力，是否需要多端，是否需要离线能力、复杂动效、系统级通知、后台任务或高性能图形。

## 六、角色关卡流程

角色不常驻，只在关卡出现。

### 6.1 产品澄清关

触发时机：

- 只有一个大致想法
- 目标用户不清楚
- MVP 边界不清楚
- 成功标准不清楚

使用：

```text
grill-me -> requirements-analysis -> docs/product.md
```

产物：

- 用户是谁
- 主要痛点
- 客户原话背后的真实问题
- MVP 必做 / 不做
- 暂缓需求和拒绝理由
- 通用产品能力，而不是单客户定制功能
- 核心流程
- 验收标准
- 风险假设
- 影响技术选型的产品约束

这一关结束后，才进入 iOS 原生、Flutter 或其他技术栈的判断。

### 6.2 架构设计关

触发时机：

- 产品边界已经基本清楚
- 准备初始化项目
- 要选技术栈
- 要拆模块和数据模型

使用：

```text
system-design -> docs/architecture.md -> docs/decisions.md
```

产物：

- 技术栈
- 模块边界
- 数据流
- 目录结构
- walking skeleton
- 关键技术决策

### 6.3 UI / UX 关

触发时机：

- 要做第一个可用界面
- 页面状态不清楚
- 想避免 AI 味 UI

使用：

```text
sleek-design-mobile-apps -> 设计稿 / 原型检查 -> 设备或模拟器验证
```

产物：

- 关键页面
- 主路径
- loading / empty / error 状态
- 桌面和移动端布局
- 视觉一致性

### 6.4 QA 验收关

触发时机：

- 每个小功能完成后
- 准备合并或进入下一轮前

使用：

```text
test-driven-development -> verification-before-completion -> docs/qa.md
```

产物：

- 测试用例
- 手动验收路径
- lint / typecheck / test / build 结果
- 剩余风险

### 6.5 上线前安全 / 运维关

触发时机：

- 准备公开部署
- 接支付、登录、用户数据、外部 API

产物：

- `.env.example`
- 密钥检查
- 错误日志
- 部署回滚方式
- 监控方案
- 基础安全检查

## 八、标准执行循环

建议把每个功能都压进这个循环：

```text
1. 需求澄清
2. 更新 product/tasks
3. 架构或设计评审
4. 实现最小切片
5. 运行 lint/typecheck/test/build
6. 前端用浏览器验证
7. 更新 qa/decisions/tasks
8. 总结偏差和下一步
```

每轮结束时问四个问题：

1. 这轮对象边界有没有变？
2. 当前状态能不能被观测到？
3. 反馈有没有真的改掉下一轮动作？
4. 系统是更稳定了，还是只是功能更多了？

如果答案不清楚，不要急着加下一个功能。

## 九、推荐启动顺序

第一次开项目时，建议这样做：

```text
第 1 轮：产品澄清
- 用 grill-me 追问想法
- 用 requirements-analysis 整理成 product.md

第 2 轮：架构骨架
- 用 system-design 出 architecture.md
- 写 decisions.md
- 明确 walking skeleton

第 3 轮：项目初始化
- 初始化代码仓库
- 建 lint/typecheck/test/build
- 建 docs/
- 建 README 和 .env.example

第 4 轮：第一条主流程
- 只做一个端到端可跑流程
- 不追求功能完整
- 先让系统稳定可验证

第 5 轮：UI/UX 和 QA 补强
- 用 sleek-design-mobile-apps 打磨移动 App 界面
- 用 test-driven-development / verification-before-completion 补验收
```

## 十、风险清单

### 10.1 一次做太大

风险：

- 偏差累积
- 反馈延迟
- 返工成本上升

控制：

- 每轮 0.5-2 天闭环
- 每次只做一个小功能或一条主流程

### 10.2 文档和代码脱节

风险：

- Codex 根据旧文档继续执行
- 你以为项目状态 A，代码实际已经到 B

控制：

- 每轮结束更新 `docs/tasks.md`
- 关键选择写入 `docs/decisions.md`
- 验收变化写入 `docs/qa.md`

### 10.3 角色太多造成噪声

风险：

- 每轮都像开会
- 输出很多，但没有产物

控制：

- 角色只在关卡调用
- 每次角色调用必须产出文件或清单

### 10.4 Skill / MCP 装太多

风险：

- 上下文变重
- 触发混乱
- 安全边界变复杂

控制：

- 第一批只装必要项
- 上线、PR、监控、设计稿等真实需求出现后再加第二批

### 10.5 没有质量指标

风险：

- 做完只能靠感觉判断
- Codex 很容易把“能跑”当“完成”

控制：

- 每个功能进入实现前写验收标准
- 每轮结束必须跑验证命令

## 十一、最终判断

这套流程适合当前场景。

原因：

- 使用者是程序员，Codex 的编码执行能力可以充分发挥。
- 当前短板主要在产品边界、架构约束、UX 状态、QA 验收和上线风险。
- 这套配置补的是控制层，不是替代编码能力。
- 它用小切片反馈降低项目失控概率，符合控制论里的闭环思路。

最终推荐采用：

```text
grill-me
-> requirements-analysis
-> system-design
-> sleek-design-mobile-apps
-> test-driven-development / verification-before-completion
-> 小切片闭环
```

不要一次做完整系统。

不要每轮调用所有角色。

不要在需求没定前写大量代码。

不要在测试没站住前追求复杂架构和性能。

先让项目稳定可跑，再让它变好。
