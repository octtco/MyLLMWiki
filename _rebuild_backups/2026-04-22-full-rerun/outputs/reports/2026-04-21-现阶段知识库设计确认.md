# 问题

现阶段真正想要的知识库，到底应该长成什么样。

## 短答案

现阶段想要的，不是一个“记了很多资料的仓库”，而是一个“以问题为入口、以方法为复用核心、能同时给人和 agent 调用的网状知识结构”。

更具体地说：

- 它不是为了存资料，而是为了在遇到问题时，能沿着稳定入口快速找到相关主题、方法、概念、案例和来源。
- 它不是纯图谱，也不是纯目录，而是“有默认入口的网状结构”。
- 它不是按学科分抽屉，而是优先按长期问题域组织顶层入口；但问题域不应预先写死，也不局限于先前举过的那五类。
- 一篇资料进入知识库后，不能只剩摘要，至少还要长出概念、方法、案例或应用方向，以及和其他资料或问题的连接。

## 展开

## 一、已经确认的设计目标

### 1. 这个知识库首先不是资料堆，而是可调用的知识结构

- 当前最核心的不满，不是“没记下来”，而是“记了很多，但真正要用的时候没有形成有效帮助”。
- 所以整理目标不能停在摘录、留痕、归档。
- 更重要的是让内容在后续理解新知识、处理问题、生成方案时，真的能被调出来。

### 2. 这个知识库既给人用，也给 agent 用

- 页面不能只追求“像文章一样好读”。
- 还要能让 agent 比较稳定地识别出：问题、方法、概念、案例、来源、连接关系。
- 目标不是只支持检索，而是支持“基于库内知识给出方案”。

### 3. 连接不是等它自己长出来，而是要被主动设计出来

- 这座知识库不是希望靠时间自然堆出联系。
- 更贴近当前需求的做法，是在整理资料时主动把概念、方法、案例和问题之间的关系设计出来。
- Obsidian 和 KG 只是当前的承载方式，不是设计本身的替代品。

## 二、当前确认的结构骨架

### 1. 整体结构应是网，而不是单一路径的树

- 一个方法可以服务多个问题。
- 一个概念可以被多个案例支撑。
- 一个案例也可能同时体现多个概念和方法。
- 单个来源未来也可能同时服务多个问题域，而不是只能塞进一个抽屉。

### 2. 即使是网，也仍然要有默认入口

- 完全无中心的图结构不利于真实使用。
- 当前更符合使用习惯的默认入口，不是抽象概念表，而是“我现在在处理什么问题”。
- 所以实际使用路径上，应优先从问题或主题进入。

### 3. 顶层入口按长期问题域组织，但问题域不预先写死

- 顶层入口更适合按长期问题域组织，而不是按“控制论 / 心理学 / 经济学”这类学科来源组织。
- 先前对“代码、赚钱、健康、人际关系、学习”的提法，更接近当前能想到的样例，不应被理解成最终固定目录。
- 更稳的原则是：问题域要根据现有资料、后续输入和真实使用需求逐步归纳。

### 4. 当前最接近目标形态的主路径

- `领域页 -> 主题页 -> 方法页 -> 概念页 / 案例页 -> 来源页`

当前这条路径里的分工，可以先这样理解：

- 领域页负责“去哪找”
- 主题页负责“我现在在处理什么问题”
- 方法页负责“可以怎么做”
- 概念页负责“怎么理解和表达”
- 案例页负责“现实里具体长什么样”
- 来源页负责“这些东西从哪来”

## 三、各类页面当前承担什么角色

### 1. 来源页是总入口和提炼起点，不是最终承载层

- 来源页要解决“这份资料到底讲了什么、为什么值得提炼”。
- 它要负责溯源，并把这份资料拆出来的概念、方法、案例、主题挂出来。
- 但它不应该把所有可复用知识都塞在自己一页里。

### 2. 主题页是默认入口，不只是路标页

- 主题页不只是导航，也不是定义大全。
- 它更像“面向怎么用的短综述页”。
- 它应该把一个问题展开成一个可进入的知识网络，而不是只挂几个链接。

### 3. 方法页是复用核心

- 当前需求里，方法不是来源页里的附属小节，也不是主题页里的顺手备注。
- 方法应该能被多个主题复用，否则一换问题场景就失去价值。
- 这也是当前设计里最需要认真扶正的一层。

### 4. 概念页负责提供理解语言和判断框架

- 概念页不是只有词典式定义。
- 它至少要回答：这是什么、怎么用大白话讲、它在原资料里怎么出现、有什么例子、能拿来干什么。
- 如果存在容易混淆或强相关概念，也应该显式连出去。

### 5. 案例页负责把抽象东西重新落回具体场景

- 案例可以是 `来源内案例`，也可以是 `现实案例`。
- 重点不在分类做多细，而在于它必须承载具体对象、具体场景、具体过程。
- 纯定义、纯结论、纯抽象判断，一般不应直接当作案例。

## 四、一篇资料整理完成后的最低产出标准

现阶段已经比较明确的一点是：一篇资料整理完成，不能只写“这篇资料讲了什么”。

最低至少要同时长出这些东西：

- 内容概览：这份资料主要讲了什么
- 概念：它拆出了哪些值得长期保留的认识单元
- 方法：它提炼出了哪些可以复用的做事方式
- 连接：它能和哪些别的资料、问题或领域发生关系
- 案例或应用方向：这些东西在哪些具体场景下可能有用

“它能解决什么问题”也很重要，但当前更像第二层组织结果，可以在上面这些产出逐渐稳定后再继续补强。

## 五、长书的当前整理原则

- 长书这类高密度来源，不适合直接从整书跳到正式知识页。
- 更合适的做法，是先做章节级中间拆解。
- 这层拆解属于提炼过程中的工作层，不等于最终知识库结构。

每一章当前至少要固定产出：

- 本章讲了什么
- 本章拆出的概念
- 本章提炼出的方法
- 本章的例子或应用方向
- 本章与其他章节或其他领域的连接

## 六、当前不要过早定死的地方

下面这些问题现在不该被伪装成“已经定稿”：

- 一篇资料的产出，长期来看是集中在一页里，还是拆成多页协同
- 方法页是否要成为仓库里的正式页面类型
- 案例是独立页、来源页的一部分，还是混合形态
- 关系主要靠 frontmatter、正文链接，还是显式关系页承载
- 章节级工作层是长期保留，还是只作为过渡层
- 主题页里的判断框架，后续更适合按什么维度展开

这里的“未定”，不是没想清楚，而是当前阶段不该过早锁死。

## 七、现阶段可直接拿来指导整理工作的几条硬要求

- 不要把知识库继续做成“资料摘录仓库”。
- 不要把顶层目录过早固化成一套看起来整齐、但后面不好扩的分类法。
- 不要只做来源摘要，必须主动往概念、方法、案例和连接上提炼。
- 不要只考虑人读起来顺不顺，也要考虑 agent 能不能稳定识别结构。
- 不要要求所有知识单元只属于一个地方，要允许多对多连接。
- 不要把还没想清楚的结构问题提前伪装成最终定稿。
