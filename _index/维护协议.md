# 维护协议

这份协议定义这座知识库怎么长，哪些内容放哪里，后续尽量按这个约定维护。

## 1. 分层

### `Clippings/`

统一收件箱。默认所有新来的原始资料先放这里，再由维护流程分流到 `raw/`、`seeds/`、`todo/` 或其他更合适的位置。

它不是长期存储层，也不是正式知识层。它更像分流前的入口区。

### `raw/`

原始资料层。文章、论文、仓库快照、数据集说明、图片、聊天记录、会议纪要、语音转写都先在这里长期保存。这个层只读，尽量保留原貌，不做知识性改写。

默认做法不再是“用户直接把资料放进 `raw/`”，而是先进入 `Clippings/`，确认类型后再迁成 `raw/` 下的标准来源包。

当前对话型材料默认统一进入 `raw/chats/`，先保原文、参与人、时间范围和引用边界，后续再提炼。

### `seeds/`

候选素材层。这里放那些值得留、但暂时还不适合直接归入 `raw/`、`wiki/`、`outputs/` 或 `todo/` 的东西。

它允许保留少量“原始内容之外的信息”，比如：

- 为什么这个东西值得记一下
- 当下的直觉判断
- 可能的后续去向

但它仍然不是正式知识层，也不是待办层。它更像分流前的稳定缓冲区。

### `wiki/`

编译层。把 `raw/` 里的资料整理成可阅读、可链接、可复用的知识页面。

- `wiki/sources/`：单个来源的摘要页
- `wiki/entities/`：具体对象页，如人、项目、产品、组织、长期关注对象
- `wiki/topics/`：围绕一个问题的主题页
- `wiki/methods/`：可复用的方法页
- `wiki/concepts/`：概念页
- `wiki/cases/`：案例页
- `wiki/domains/`：领域页
- `wiki/relations/`：关系页
- `wiki/_templates/`：模板

### `todo/`

行动层。这里不负责定义概念，也不负责长期来源沉淀，而是负责当前要推进什么。

- `todo/goals/`：目标页，负责方向、阶段、卡点和推进策略
- `todo/tasks/`：待办页，负责当前可执行动作
- `todo/_templates/`：行动层模板

### `outputs/`

派生产出层。问答、综述、slide、图表说明、阶段性分析都放这里。长书、长报告和高密度长材料的工作层，也先放这里。它们可以在确认后回流进 `wiki/`。

### `assets/`

跨页面复用的图片、附件、导出文件。

### `_index/`

索引、队列、规则。这里是整个库的控制面。

## 2. 命名规则

### 收件箱内容

`Clippings/` 里的文件名不要求第一次就标准化。更重要的是先保原文、先进库、先别误删。

标准命名和来源包结构，留到正式分流时再做。

### 原始资料

`raw/` 里的内容默认视为已导入完成的原始来源，不再人工整理、不补总结、不改文件内容。

原始资料优先采用“来源包”形式，也就是一个来源一个目录，把正文、图片、附件、元数据放在一起。优先级高于“所有文件平铺在类型目录里”。

建议格式：

- 网页：`raw/web/<来源包>/<简短主文件名>.md`
- 聊天 / 会议 / 转写：`raw/chats/<来源包>/transcript.md`
- 论文：`raw/papers/<来源包>/paper.pdf`
- 仓库：`raw/repos/<来源包>/`
- 数据集：`raw/datasets/<来源包>/`
- 独立图片来源：`raw/images/<来源包>/`

来源包内建议结构：

- 正文或主文件：`<简短主文件名>.md` / `transcript.md` / `paper.pdf` / `README.md`
- 配套图片：`images/` 或 `attachments/`
- 配套附件：`files/`
- 来源元数据：`metadata.json` 或 `metadata.md`

原则：

1. 正文和配图尽量放在同一个来源包里
2. 来自某个来源的图片，不单独抽到通用图片目录
3. 只有“本身就是独立来源”的图片，才单独放进 `raw/images/`
4. `raw/` 中如已存在旧结构，先兼容，不强制回迁
5. 网页、聊天或其他文本类来源，正文主文件默认使用简短直观的语义名；只有在容易撞名时才补最少的消歧尾巴

### 编译页

- 来源页：`来源名 - 标题.md`
- 对象页：`对象名.md`
- 主题页：`主题名.md`
- 方法页：`方法名.md`
- 概念页：`概念名.md`
- 案例页：`案例名.md`
- 领域页：`领域名.md`
- 关系页：`概念A - 关系类型 - 概念B.md` 或 `关系名.md`

### 行动页

- 目标页：`目标名.md`
- 待办页：`YYYY-MM-DD-事项名.md` 或 `事项名.md`

### 输出页

建议格式：

- `YYYY-MM-DD 问答 - 主题.md`
- `YYYY-MM-DD 综述 - 主题.md`
- `YYYY-MM-DD 幻灯片 - 主题.md`
- `YYYY-MM-DD 图表说明 - 主题.md`

## 3. 页面职责

### 来源页

解决“这份资料到底讲了什么、当前支持什么、原始材料在哪”。重点是摘要、证据、边界，以及已抽出的对象、主题、方法、概念和案例。

来源页状态约定：

- `active`：知识网络的活跃节点，被两个以上下游页面（主题 / 方法 / 概念 / 案例 / 对象 / 关系页）实际引用，或已经过真实使用 / 交叉验证
- `review`：已存档、按需取用的正常终态，不是等待处理的中转状态；它表示“已按来源页标准保存，但尚未被下游充分引用或验证”
- `review` 不要求毕业。当某个来源后续被两个以上下游页面引用，或被真实用过一次，再顺手提成 `active`

### 对象页

解决“这个具体对象现在是什么、为什么重要、最近发生了什么、和什么强相关”。重点是当前状态，不是抽象定义。

### 主题页

解决“围绕某个问题，当前库里已经知道什么”。重点是局部组织，不是目录，也不是定义大全。

### 方法页

解决“什么时候该调用这条方法，它在修什么误判，它真正补了什么动作或判断”。重点是调用时机，不是态度句，也不是纯步骤表。

### 概念页

解决“某个概念在整个库里最小指什么”。重点是定义、边界、接口、相关来源，以及哪些主题或方法会调用它。

概念页尽量承载单个原子概念。像“Spec 和 Skill 的协同”这类复合表达，优先写成关系说明或主题页里的组织语句。

### 案例页

解决“一个抽象方法或概念在具体对象、具体场景、具体过程中到底怎样发生”。重点是对象、场景和过程，不能只剩结论。

### 领域页

解决“这座知识库当前把哪些长期问题放在同一个入口下维护”。重点是导航和扩展边界，不是冒充完整学科目录。

### 关系页

解决“两个概念、来源或结构之间，到底是什么关系，以及这条关系为什么值得单独说明”。重点是关系定义、证据基础、适用边界、限制条件。

严格 KG 笔记法里，关系页是正式页面类型。但不是每条边都要建页，只有在这条关系本身值得持续补证据、反复复用、或者容易被误解时，才单独建关系页。

### 目标页

解决“为什么现在要做这件事、做到什么算完成、当前卡点是什么、下一步怎么推进”。重点是行动控制，不是知识总论。

### 待办页

解决“当前马上要做什么”。重点是执行动作，不冒充方法页。

### 输出页

解决“针对具体问题，这次产出了什么”。它可以是一次问答，也可以是一次报告、slide、图表，或者长材料拆解过程中的章节工作层和候选池。它属于任务结果层，不等于已经沉淀进知识库本体。

## 4. 维护原则

1. `Clippings/` 是默认统一收件箱，先接住新材料，再分流
2. 原始层和编译层分开，`raw/` 只读，不把总结写回 `raw/`
3. `seeds/` 用来接住暂未定型但值得保留的候选素材，不强求第一次落盘就分类正确
4. 每个重要结论尽量能追到来源
5. 聊天、会议、私聊和转写这类对话型材料，先保原文和语境，再提炼
6. 正式整理一般先有来源页，再看是否需要更新对象页、目标页，最后再往下筛主题、方法、概念和案例
7. 长书、长报告和其他高密度长材料，先在 `outputs/` 建工作层，不直接从整本材料跳到正式页
8. 有价值的问答结果和规程样板先落到 `outputs/`，稳定后再提炼回 `wiki/`
9. 新页面尽量挂到索引，不制造孤页
10. 主题页、方法页、概念页、案例页、对象页、目标页各自守住自己的职责，不偷彼此的活
11. 正文优先，链接尽量嵌在第一次真正需要理解的位置，不把解释主要推到文末
12. 对象页承接人、项目、产品、组织这类具体对象，不拿概念页替代
13. 目标和待办优先放在行动层维护，不拿主题页替代
14. 严格 KG 维护时，要始终有“关系页意识”；即使当前证据不足，也要区分“暂不建关系页”和“这不是关系页候选”这两件事
15. 一般任务仍按“先给方案、确认后执行”的协作方式处理
16. 对知识库维护任务，默认允许先执行后汇报；仅在高风险、破坏性、或存在重大歧义时再确认

### 入库分级（2026-06-10 起）

不是每个新来源都值得全套来源包加来源页。新仓库和新链接类线索默认走轻量档：

1. 新 GitHub 仓库、工具、产品类线索，默认先登记进 [[样本池]]：一行一条，链接 + 一句话定位 + 日期
2. 满足以下任一条件时，才升全套 `raw/` 来源包 + 来源页：
   - 维护者真正读过主要内容（不只是 README 扫一眼）
   - 真正用过或跑过
   - 它被某个主题页 / 方法页的论证实际需要
3. 用户明确要求完整入库时，按要求执行
4. 网页文章、聊天记录、书籍、论文等内容本身就是材料的来源，仍走原有完整流程
5. 已有的存量来源页不做回溯降级，原样保留

### 验证缺口的记录位（2026-06-10 起）

1. 来源页的 `当前边界` 一节是验证缺口的第一记录位，入库时照常写
2. 不再为每个新来源在 [[待处理]] 自动生成“补交叉来源 / 跑验证 / 复核边界”类待办
3. [[待处理]] 只收维护者主动决定要做的事
4. 存量验证类事项归档在 [[验证债]]，按需取用，不要求清零

### 知识库维护任务的默认范围

这里的“知识库维护任务”包括但不限于：

1. `Clippings/` 收件箱的读取、识别、分流与清理
2. `seeds/` 候选素材的落盘、整理与分流
3. `raw/web/`、`raw/chats/` 等原始来源包的规范化
4. 远程图片本地化
5. `wiki/sources/`、`wiki/entities/`、`wiki/topics/`、`wiki/methods/`、`wiki/concepts/`、`wiki/cases/`、`wiki/domains/`、`wiki/relations/` 的增量生成与更新
6. `todo/goals/`、`todo/tasks/` 与知识库主线直接相关的增量维护
7. `outputs/` 的落盘与必要回写
8. `_index/` 下的索引、待处理、日志、健康检查维护

### 知识库维护任务中需要再次确认的情况

只有遇到这些情况时，再停下来确认：

1. 需要删除、覆盖、合并已有知识页或批量重命名页面
2. 需要重构目录结构或迁移大量历史内容
3. 证据不足，但要输出强结论或大范围改写既有结论
4. 需要引入仓库外的新资料，且这会明显改变当前结论
5. 存在多个代价差异明显的路径，且当前无法合理默认其中一个

补充说明：

- `Clippings/` 中的原文件在成功迁入正式落点并核对无误后，默认允许删除，不单独视为破坏性操作

## 5. 推荐工作流

1. 把新材料先放进 `Clippings/`
2. 先判断它是原始来源、未成熟种子、行动项，还是只是当前任务的临时输入
3. 明确是原始来源的，迁到 `raw/` 对应目录
4. 暂时不适合直接归入其他层的内容先放进 `seeds/inbox/`
5. 如果内容主要是行动方向，先更新 `todo/goals/`；如果主要是执行动作，先更新 `todo/tasks/`
6. 先判断任务类型：短资料、长资料、对话型来源、对象更新、目标推进、多来源同主题、还是只做输出
7. 只有需要继续处理的项，再在 [[待处理]] 里登记
8. 先为资料生成或更新来源页
9. 如果稳定信息主要落在具体人 / 项目 / 产品上，先更新对象页
10. 再按 `主题页 -> 方法页 -> 概念页 / 案例页` 的顺序筛正式抽象页
11. 先补 frontmatter 邻接和 `## 关系` 里的轻量关系；证据够时再单独建关系页
12. 有新的分析结果、规程样板或任务结果时写进 `outputs/`
13. 确认有长期价值后，再合并回 `wiki/`
