# 维护协议

这份协议定义这座知识库怎么长，哪些内容放哪里，后续尽量按这个约定维护。

## 1. 分层

### `raw/`

原始资料层。文章、论文、仓库快照、数据集说明、图片都先放这里。这个层只读，尽量保留原貌，不做知识性改写。

仓库根目录下的 `Clippings/` 视为网页剪藏导入收件箱，不视为长期存储层。进入知识库前，应先把其中的文件规范化为 `raw/web/` 下的标准来源包，再删除 `Clippings/` 中对应原文件。

### `seeds/`

候选素材层。这里放那些值得留、但暂时还不适合直接归入 `raw/`、`wiki/`、`outputs/` 或 `todo/` 的东西。

它允许保留少量“原始内容之外的信息”，比如：

- 为什么这个东西值得记一下
- 当下的直觉判断
- 可能的后续去向

但它仍然不是正式知识层，也不是待办层。它更像分流前的稳定缓冲区。

### `wiki/`

编译层。把 `raw/` 里的资料整理成可阅读、可链接、可复用的知识页面。

- `wiki/sources/`：单个来源的摘要页
- `wiki/concepts/`：概念页
- `wiki/relations/`：关系页
- `wiki/topics/`：主题页
- `wiki/_templates/`：模板

### `outputs/`

派生产出层。问答、综述、slide、图表说明、阶段性分析都放这里。它们可以在确认后回流进 `wiki/`。

### `assets/`

跨页面复用的图片、附件、导出文件。

### `_index/`

索引、队列、规则。这里是整个库的控制面。

## 2. 命名规则

### 原始资料

`raw/` 里的内容默认视为已导入完成的原始来源，不再人工整理、不补总结、不改文件内容。

原始资料优先采用“来源包”形式，也就是一个来源一个目录，把正文、图片、附件、元数据放在一起。优先级高于“所有文件平铺在类型目录里”。

建议格式：

- 网页：`raw/web/<来源包>/<简短主文件名>.md`
- 论文：`raw/papers/<来源包>/paper.pdf`
- 仓库：`raw/repos/<来源包>/`
- 数据集：`raw/datasets/<来源包>/`
- 独立图片来源：`raw/images/<来源包>/`

来源包内建议结构：

- 正文或主文件：`<简短主文件名>.md` / `paper.pdf` / `README.md`
- 配套图片：`images/`
- 配套附件：`files/`
- 来源元数据：`metadata.json` 或 `metadata.md`

原则：

1. 网页正文和网页配图尽量放在同一个来源包里
2. 来自某个来源的图片，不单独抽到通用图片目录
3. 只有“本身就是独立来源”的图片，才单独放进 `raw/images/`
4. `raw/` 中如已存在旧结构，先兼容，不强制回迁
5. 网页或其他文本类来源，正文主文件默认使用简短直观的语义名；只有在容易撞名时才补最少的消歧尾巴

### 编译页

- 来源页：`来源名 - 标题.md`
- 概念页：`概念名.md`
- 关系页：`概念A - 关系类型 - 概念B.md` 或 `关系名.md`
- 主题页：`主题名.md`

### 输出页

建议格式：

- `YYYY-MM-DD 问答 - 主题.md`
- `YYYY-MM-DD 综述 - 主题.md`
- `YYYY-MM-DD 幻灯片 - 主题.md`
- `YYYY-MM-DD 图表说明 - 主题.md`

## 3. 页面职责

### 来源页

解决“这份资料到底讲了什么”。重点是摘要、关键点、证据、相关概念、相关主题。

### 概念页

解决“某个概念在整个库里是什么意思”。重点是定义、边界、争议点、相关来源。

概念页尽量承载单个原子概念。像“Spec 和 Skill 的协同”这类复合表达，优先写成关系说明或主题页里的组织语句。

### 关系页

解决“两个概念、来源或结构之间，到底是什么关系，以及这条关系为什么值得单独说明”。重点是关系定义、证据基础、适用边界、限制条件。

严格 KG 笔记法里，关系页是正式页面类型。但不是每条边都要建页，只有在这条关系本身值得持续补证据、反复复用、或者容易被误解时，才单独建关系页。

### 主题页

解决“围绕某个问题，当前库里已经知道什么”。重点是整合，不是重复摘抄。

### 输出页

解决“针对具体问题，这次产出了什么”。它可以是一次问答，也可以是一次报告、slide 或图表。它属于任务结果层，不等于已经沉淀进知识库本体。

## 4. 维护原则

1. 原始层和编译层分开，`raw/` 只读，不把总结写回 `raw/`
2. `seeds/` 用来接住暂未定型但值得保留的候选素材，不强求第一次落盘就分类正确
3. 每个重要结论尽量能追到来源
4. 优先增量更新，不全量重写
5. 有价值的问答结果要回写，不让思考只留在对话里
6. 新页面尽量挂到索引，不制造孤页
7. `outputs/` 可以先行承接阶段性结果，稳定后再提炼回 `wiki/`
8. 严格 KG 维护时，要始终有“关系页意识”；即使当前证据不足，也要区分“暂不建关系页”和“这不是关系页候选”这两件事
9. 一般任务仍按“先给方案、确认后执行”的协作方式处理
10. 对知识库维护任务，默认允许先执行后汇报；仅在高风险、破坏性、或存在重大歧义时再确认

### 知识库维护任务的默认范围

这里的“知识库维护任务”包括但不限于：

1. `seeds/` 候选素材的落盘、整理与分流
2. 来源包规范化
3. 远程图片本地化
4. `Clippings/` 收件箱到 `raw/web/` 标准来源包的迁移
5. `wiki/sources/`、`wiki/concepts/`、`wiki/relations/`、`wiki/topics/` 的增量生成与更新
6. `outputs/` 的落盘与必要回写
7. `_index/` 下的索引、待处理、日志、健康检查维护

### 知识库维护任务中需要再次确认的情况

只有遇到这些情况时，再停下来确认：

1. 需要删除、覆盖、合并已有知识页或批量重命名页面
2. 需要重构目录结构或迁移大量历史内容
3. 证据不足，但要输出强结论或大范围改写既有结论
4. 需要引入仓库外的新资料，且这会明显改变当前结论
5. 存在多个代价差异明显的路径，且当前无法合理默认其中一个

补充说明：

- `Clippings/` 中的原文件在成功迁入 `raw/web/<来源包>/` 后默认允许删除，不单独视为破坏性操作

## 5. 推荐工作流

1. 把新资料放进 `raw/`
2. 暂时不适合直接归入其他层的内容先放进 `seeds/inbox/`
3. 只有需要继续处理的项，再在 [[待处理]] 里登记
4. 为资料生成来源页
5. 从多份来源页里抽概念
6. 先补 frontmatter 邻接和 `## 关系` 里的轻量关系；证据够时再单独建关系页
7. 从概念、关系和来源页里组织主题页
8. 有新的分析结果时写进 `outputs/`
9. 确认有长期价值后，再合并回 `wiki/`
