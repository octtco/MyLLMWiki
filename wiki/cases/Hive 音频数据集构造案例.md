# Hive 音频数据集构造案例

## 具体对象

这个案例的对象是 [[Hive]]：一个面向 query-based universal sound separation 的合成音频数据集和数据构造管线。

## 具体场景

Hive 面对的问题不是“音频数据不够大”这么简单，而是野外音频天然有弱标签、背景噪声和多事件共现。模型如果直接在这类数据上训练，可能学到的是目标声源和背景事件的伪相关，而不是干净的分离关系。

在这个场景里，真正需要控制的是监督信号：查询、目标声源、混合背景和标签之间的关系是不是足够纯。

## 具体过程

Hive 的路线是先从非受控录音里筛出高纯度单事件片段，再按语义一致的规则合成混合音频。

项目页和论文入口把问题界定为监督纯度不足，仓库 README 则把工程形态补得更细：它记录了数据规模、metadata-only 与预生成混合音频两种发布形态，以及切分音频、单标签过滤、事件过滤、标注、分类和超分等管线步骤。

这条路线把模型训练入口往前推：先整理数据构造和发布方式，再谈 AudioSep-hive、FlowSep-hive 这类推理入口和模型效果。

## 这个案例说明了什么

这个案例说明 [[机器学习工程入口]] 里“模型之前先看数据”的判断，在音频分离任务里会具体变成“先看监督信号是否纯净”。

它也支撑 [[音频分离数据集与监督纯度]]：同样是扩大数据集，Hive 的重点不是单纯堆小时数，而是通过数据筛选和合成协议改变训练样本的监督关系。

## 边界

- 当前知识库只保存项目页、README 和 arXiv 入口，不下载 Hive-ALL 数据集。
