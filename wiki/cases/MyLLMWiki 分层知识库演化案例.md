# MyLLMWiki 分层知识库演化案例

## 具体对象

这个案例的对象是 [[MyLLMWiki]]：一座基于 Obsidian 的个人 LLM 知识库。它已经不是普通文件夹，而是带有 `Clippings / raw / seeds / wiki / todo / outputs / assets / _index` 分层、维护协议、索引和 skill 支撑的持续整理系统。

## 具体场景

这个案例发生在个人知识库从“资料收集”转向“可持续知识网络”的阶段。用户希望网页、书、聊天记录、灵感、目标和任务都能进入同一套库，但又不能把原始材料、临时判断、正式知识和行动控制混成一锅。

核心约束有三条：

- 原始资料要能追溯，不能被总结污染。
- 抽象页要能复用，不能只是来源摘要换皮。
- 维护过程要能继续跑，不依赖一次性大整理。

## 具体过程

最早的入口是外部范式和结构讨论，例如 Karpathy LLM Wiki、KG、PARA、IRK 等参照。但这些参照没有被照搬，而是被真实材料反复压测。

第一步先把 `raw/` 定成来源保存层，把来源页作为正式知识层的入口。这样每个结论先能回到原始网页、书籍、仓库或聊天记录。

第二步加入 `outputs/` 工作层。像《工程控制论》这种长材料，不能从整本书直接跳到概念页和方法页，所以先在 `outputs/reports/` 里做范围、章节工作卡、候选池和终筛，再回写到 `wiki/`。

第三步把 `Clippings/` 和 `seeds/` 接进来。前者负责先接住新材料，后者负责保存暂不稳定但值得保留的线索，避免为了清空入口硬把高波动内容写成正式知识页。

第四步把聊天归档切成项目级来源。[[Codex rollout - MyLLMWiki 项目切片（2026-04-13 至 2026-04-23）]] 让这座库自己的演化过程也变成可追溯来源，并进一步回写到对象页和目标页。

## 这个案例说明了什么

这个案例说明：个人知识库的结构不是先设计完再使用，而是在真实材料、长文、网页、聊天和行动目标的压力下逐步长出来。

它也说明 [[Skill]] 在知识库维护里不只是“会做某个任务”，还可以承载路由规则、页型边界、写作口径和审查门槛。这里最关键的不是目录多，而是不同层各守职责：来源保真、输出承接工作过程、正式页沉淀复用结构，索引和日志维持可发现性。

## 边界

- 这个案例当前只说明 MyLLMWiki 这座库的演化路线，不等于所有个人知识库都应该采用同样目录。
- 聊天归档里的很多判断仍强依赖当时上下文，不适合直接抽成通用方法。
- 这不是一个完成态案例，更像一个正在运行中的维护系统样本。
