# 代码库图谱化理解

## 这页到底在回答什么

这页先回答一个很窄的问题：当代码库、知识库或网页交互过程太大，agent 和人怎样先获得可探索的结构入口，而不是每次都从零开始 grep。

## 为什么这个问题会让人反复做散

大家谈代码理解时，很容易只谈“模型上下文够不够长”或“搜索工具够不够快”。但真实问题常常不是读不到文件，而是缺少一个稳定的地图：哪些文件形成模块、哪些路径承载业务逻辑、哪些交互过程能复盘。

## 当前最稳的主线

当前这组三个来源支持一条保守主线：代码理解可以分成静态结构索引和动态过程记录两层。[[GitHub - Understand Anything]] 和 [[GitHub - CodeGraph]] 代表静态图谱 / 预索引路线，[[GitHub - rrweb]] 则提醒我们，前端交互本身也可以被录制、回放、复盘。

这条线和 [[AI coding framework]] 的关系是：framework 不只需要 spec、skill 和 task，也需要更好的上下文供给层。图谱和回放不是替代阅读源码，而是降低第一次进入系统时的盲搜成本。

## 这条主线怎样一段一段展开

第一段是从“读文件”转向“看结构”。[[GitHub - Understand Anything]] 把 codebase、knowledge base 和 docs 转成 interactive knowledge graph，重点是让人能 explore、search、ask questions。

第二段是把结构服务给 agent。[[GitHub - CodeGraph]] 更明确面向 Claude Code、Codex、Cursor 等 coding agent，主张本地预索引能减少 token 和工具调用。这个说法还需要复测，但它指向的需求很清楚：agent 需要少走弯路。

第三段是补上动态证据。[[GitHub - rrweb]] 不是代码图谱工具，但它把网页交互录制和回放变成可观察材料。对前端项目和浏览器 agent 来说，动态过程有时比静态文件结构更接近真实问题。

## 当前边界

- 当前只整理 README 入口，没有运行真实项目分析。
- README 里的效率收益和 benchmark 口径未复测。
- 图谱和回放是定位与理解辅助，不等于自动证明代码修改正确。

## 后续可追的问题

- 选一个本地项目试跑 Understand Anything 或 CodeGraph，记录是否真的减少探索成本。
- 结合前端调试任务评估 rrweb 这类 session replay 能否给 agent 提供稳定证据。
