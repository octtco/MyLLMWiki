# 系统设计学习入口

## 这页真正想回答什么

这页想回答的是：当一个人想补系统设计时，怎样先抓住学习入口，而不是立刻淹在数据库、缓存、负载均衡、CAP、消息队列这些名词里。

## 为什么这个问题容易散

系统设计的材料天然碎。每个组件都能展开成一门课，每个面试题又能拆出容量估算、读写比例、可用性、延迟、数据模型和部署约束。如果一开始只按名词收藏资料，很快会得到一张很大的清单，但仍不知道下一次遇到开放题该怎么组织答案。

## 当前最稳的主线

[[GitHub - system-design-primer]] 当前给出的入口不是“背完所有组件”，而是先承认系统设计是一组取舍。README 反复把 trade-off 放在中心位置：性能和可扩展性、延迟和吞吐、可用性和一致性都不是孤立知识点，而是设计讨论里的压力来源。

顺着这条线，学习入口可以先分成两层。第一层是组件地图：知道 DNS、CDN、负载均衡、数据库、缓存、异步队列、通信协议和安全大致解决什么问题。第二层是表达过程：先问 use cases、constraints、assumptions，再画 high level design，接着钻 core components，最后根据瓶颈 scale design。

这样系统设计学习就不只是“补知识点”，而是在训练一种对话顺序：先把问题框住，再让组件选择服务于约束。它和 [[控制论式系统思维]] 有一点相邻气质，都在提醒人不要直接跳到优化动作；但这里的主场更窄，主要服务软件系统学习和面试表达。

[[GitHub - developer-roadmap]] 给这页补的是导航层。它不替代 system-design-primer 里的设计讨论，但提供了 system design roadmap、software architect roadmap、API design roadmap 等入口，让学习者能先知道系统设计和相邻能力在开发者成长地图里的位置。

## 当前边界

- 当前主要由 system-design-primer 和 developer-roadmap 支撑，先不拆系统设计概念页。
- 这页不把面试训练等同于真实工程经验。
- README 中的外链和题解本轮没有逐条验证，后续若要长期学习，应再补官方文档、工程博客和真实架构复盘。
