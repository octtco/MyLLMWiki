# 设计系统与平台规范

## 这页到底在回答什么

这页先回答一个窄问题：做产品界面时，哪些设计资料只是灵感，哪些已经带有团队或平台约束。

## 为什么这个问题会让人反复做散

设计系统很容易被看成“好看的组件库合集”。这样会漏掉两个关键差异：一类资料是组织内部或开源社区沉淀出来的产品建造规则，另一类资料是平台方给出的体验边界。前者适合横向找样本，后者在具体平台上更像约束条件。

## 当前最稳的主线

当前两份来源刚好提供了两个入口。[[GitHub - awesome-design-systems]] 负责横向样本池：它把组件、语气文案、设计师套件和源码开放程度放在一张表里，提醒我们设计系统不止是 UI 控件。[[Apple - Human Interface Guidelines]] 负责平台规范：它不是普通参考站，而是 Apple 平台上关于基础元素、模式、组件、输入和技术集成的官方设计入口。

[[GitHub - Impeccable]] 补的是第三个入口：当设计判断进入 AI coding 环境时，规范不只来自平台文档或设计系统样本，也可能被打包成 skill、commands 和 anti-pattern rules，让 agent 在生成、审查、打磨界面时有更具体的设计语言。

## 这条主线怎样一段一段展开

第一段是先区分“样本池”和“约束源”。awesome list 适合用来找竞品、找不同组织如何表达设计规则；HIG 则适合在做 Apple 平台体验时检查自己是否顺着平台惯例走。

第二段是把设计系统的构成拆开。组件只是其中一块，Voice & Tone、Designers Kit、Source code 也会影响团队能不能把设计真正落进产品。

第三段是把平台规范放回产品判断。Apple HIG 首页强调 Hierarchy、Harmony、Consistency，并把页面结构拆成 Foundations、Patterns、Components、Inputs、Technologies，这意味着界面判断要同时看视觉层级、设备软硬件关系、系统组件和输入方式。

第四段是保持证据边界。awesome list 的表格标记不能替代逐项验证；HIG 首页也不能替代每个组件子页。当前这页先做入口组织，后续有具体项目或界面问题时，再进入更细的组件或模式来源。

第五段是把 agent 设计纪律和传统设计规范分开。Impeccable 里的 anti-pattern 适合提醒 AI 前端常见问题，但它不是平台官方规范，也不是完整产品设计系统。它更适合放在“AI 生成界面如何少犯默认错误”这一侧。

## 当前边界

- 当前只引入了一个社区导航表和一个 Apple 官方规范入口，暂不扩写成完整设计系统方法论。
- Impeccable 当前只作为 AI 前端设计 skill 样本保存，未验证其 CLI、浏览器扩展和真实项目改进效果。
- 不把 HIG 首页上的 Liquid Glass 线索直接写成具体 UI 规则；它需要单独读对应官方文档。
- 不把 awesome list 里的所有系统都视为仍然活跃或质量相同。

## 后续可追的问题

- 是否需要单独整理 Material Design、Fluent、Carbon、Polaris、Ant Design 等代表性系统，补成横向比较。
- 如果后续做 iOS / macOS / visionOS 产品页，可把 HIG 子页面拆成更具体的组件和模式来源。
- 是否需要从这些来源中筛出“设计系统成熟度检查表”。
- 是否需要单独整理“AI 前端设计反模式检查表”，并区分平台规范、设计系统和 agent skill 的边界。
