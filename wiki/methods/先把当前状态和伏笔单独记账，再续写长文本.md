# 先把当前状态和伏笔单独记账，再续写长文本

## 这页在解决什么卡点

长文本一写长，最容易出的不是文笔问题，而是“前面讲过什么、人物现在在哪、哪些坑还没填、哪些设定已经变了”开始一点点失控。

## 为什么这里最容易做错

最常见的错法有两个：一个是指望模型自己记住一切，另一个是每次都把几百章全塞回去重读。前者很快会失忆，后者又会把上下文拖爆，还不一定真抓得住当前最关键的状态。

## 这一步到底在做什么

这一步真正补的，是把“当前到底写到哪了”从正文里拎出来，单独做成可读的状态快照。也就是说，你不再把“当前关系、当前伤势、当前线索、当前伏笔、当前资源、最近发生了什么”埋在海量正文里等模型自己找，而是主动把它们记成一层外置账本。这样模型续写时，先看的是当前 [[状态]]，不是整片海。

更稳一点说，这一步不是“多做几张表”，而是把续写入口从“回看全部历史”改成“先读当前工作窗口”。它真正外置的，是这轮继续写下去最需要知道的那层东西。

## 最小动作

- 先把“当前局面”拆出来，做成一张当前状态卡。
- 再把容易丢的长线东西单独记账，比如伏笔、人物状态、资源变化、章节结算。
- 续写前只补读最近相关章节，不机械追满全部历史。
- 如果本轮正文改写了关键状态，先回写状态卡和账本，再继续下一轮。
- 把这些外置文件和 [[Skill]]、大纲、设定一起配合，而不是彼此脱节。
- 让“状态卡 -> 最近变化 -> 相关原文”形成固定读取顺序。

## 到什么时候该调用它

- 文本已经长到“光靠记忆和搜索没法稳住”的时候。
- 每次续写前，你都要先花很久解释当前局面。
- 人物关系、伤势、资源、伏笔开始出现前后矛盾。
- 你发现真正要找的，不是第 37 章里某句话，而是“当前整体状态是什么”。

## 最小案例

- 对象：长篇网文续写。
- 当时卡在哪：章节越写越多，模型既容易忘，也不适合每轮全量重读。
- 这条方法具体先做了什么：先读当前状态卡，再补最近章节，再看设定索引和伏笔账本，而不是机械吞全部正文。
- 结果怎么变了：续写的入口从“海量重读”改成了“读当前状态 + 读最近变化”，上下文更稳，也更省窗口。

## 已出现的具体用法

- [[Linux.do - AI 辅助小说写作赚钱方式（第 2 期）]]：明确建议补“待回收伏笔”“主角表”“写作记录表”“关键设定”“人物状态”。
- [[Linux.do - 小说长上下文续写与状态维护]]：把“当前状态卡 -> 最近章节 -> 设定总索引 -> 伏笔账本”的读取顺序讲成了完整方法。 
- [[MyLLMWiki]]：这座知识库的 `[[总览]]`、`[[待处理]]`、`[[log|运行日志]]` 其实也在干同一件事——先把项目当前状态外置，再决定下一轮该读哪部分历史和该做哪一步。 

## 不同场景里会怎么变形

- 在小说里，它会长成状态卡、伏笔表、人物状态表和章节结算表。
- 在知识库维护里，它会长成总览、队列、日志和健康检查。
- 在长项目推进里，它会长成阶段状态、风险清单和最近变更记录。

## 边界

- 它不等于把所有细节都再抄一遍。
- 它也不等于永远不读原文；只是先读当前状态，再决定回读窗口。
- 如果内容本来就很短，这套账本化管理也没必要硬上。
- 它解决的是长文本续写失控，不解决设定本身是否成立。
