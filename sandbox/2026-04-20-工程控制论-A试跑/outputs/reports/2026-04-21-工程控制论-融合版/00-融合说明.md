# 工程控制论：1-2 章融合说明

## 这套文件在解决什么问题

这套“融合版”不是第三套平行笔记，而是专门拿来修正前两套各自的问题。

- [[2026-04-21-工程控制论-旧结构对照/01-第一章-章节拆解-旧结构版|旧结构版]] 的优点，是结构感比较强，知道一章里能拆出什么概念、方法和主题。
- [[2026-04-21-工程控制论-PDF正文重跑/01-第一章-章节工作页-基于PDF正文|PDF 正文重跑版]] 的优点，是判断能直接回到原书页码。
- 但前者容易“像是懂了，实际上证据不够硬”，后者又容易“虽然稳，但太薄，拆不出东西”。

所以这一版的原则很简单：

1. 用旧结构版的“拆解意识”做骨架。
2. 用 PDF 正文抽取做证据底座。
3. 每个重要判断都尽量标出页码线索。
4. 明确区分“原书直接支持”和“基于原书的低强度归纳”。

## 这版该怎么看

建议这样看：

1. 先看“这一章到底在干什么”，确认本章在全书里的位置。
2. 再看“按小节拆开看”，确认每节到底贡献了什么。
3. 再看“概念候选”和“方法候选”，判断哪些值得后续升格成正式知识页。
4. 最后看“对你 / 对 agent 的直接用途”，判断它是不是只停留在书本层。

## 证据标注约定

### 1. `直接支持`

表示这个判断基本能直接从该页正文读出来，最多只做很轻的整理。

示例写法：

- `p26：明确说“通常首先要求的就是稳定性”。`
- `p35：明确列出模型建立的三种常用方法：解析法、实验测试法、统计试验法。`

### 2. `低强度归纳`

表示这不是原书原句，而是根据章节结构、段落安排和若干页正文，整理出来的“比较稳的理解”。

这类内容允许保留，但不会直接当成正式概念页结论。

### 3. 引用方式

- 以原书 PDF 物理页码为准，例如 `p31-p32`。
- 只保留很短的原句句柄，目的是让你知道我是从哪里得出判断，不追求长摘录。
- 如果某个判断主要来自多页合看，就写成页段，例如 `p35-p37`。

## 这版与正式知识库的关系

当前这套文件仍然只是 `outputs/` 里的章节工作页，不是 `wiki/` 正式页。

它的作用是先回答两个问题：

1. 这本长书到底能拆出哪些真正有复用价值的概念、方法和主题？
2. 哪些判断已经足够稳，可以继续往正式知识页推进？

所以这版会故意保留一些“候选”状态，比如：

- 某个概念已经值得拆，但还没决定要不要正式建页。
- 某个方法已经能看见雏形，但还需要后续章节继续压实。

## 这版额外对齐的知识库约束

这一轮不只参考旧结构版和 PDF 正文，还显式对齐了 [[2026-04-20-知识库需求澄清记录]] 里已经说清的几条要求：

- 长书要先有“章节级中间拆解层”，不能直接从整书跳正式知识页。
- 章节工作页不能只回答“这章讲了什么”，还要长出概念、方法、例子 / 应用方向、跨章节 / 跨领域连接。
- 概念不能只给抽象定义，还要有大白话解释、原书出现方式、例子、用途。
- 方法不能只给口号，还要说明适用于什么情况、原书怎么支撑、可迁移到哪里。

所以融合版的章节页，除了正文拆解外，还会强制补一组“按知识库工作层回收”的栏目，方便后面继续拆页。

## 这一轮最重要的判断标准

这轮先别急着问“能不能一口气进 wiki”。

更重要的是看这版有没有同时做到四件事：

- 比 PDF 重跑版更细，真的拆出了概念和方法。
- 比旧结构版更稳，每个关键判断都能回到原书页码。
- 比以前更适合后续迁到你的知识库体系里。
- 不是只像读书笔记，而是真能给你和 agent 提供后续判断抓手。
